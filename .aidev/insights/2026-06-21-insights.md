## 再発パターン

### 追跡: 前回 ★最重要「上流での自リポジトリ確認不足」

- **再発せず**。`aidev-worktree` は外部知識でなく**自リポジトリの事実（`.gitignore`・`git ls-files`・既存 CLI 挙動）を
  research で直読・実機検証**してから設計した。前回の教訓の方向（上流で内部仕様を読む）が実作業で実践された。
- ただし AGENTS.md への一般則化（前回 ★最優先提案）の反映状況は本データだけでは判別不可（次アクション参照）。

### 新規パターン（ハーネス＝sh/ps1 二本立て CLI ドメイン固有）

`aidev-worktree`（decisions D2/D4・review must）で顕在化した、**前回ドメインには無かった**実装言語起因の罠群:

- **① 二言語 CLI のランタイム検証の穴**（systemic）: 開発環境に **pwsh が無く `aidev.ps1` が未検証のまま着地**しうる
  （#27）。review はコード照合で must を捕捉できたが、構造照合頼みは脆い。
- **② `set -eu` / PowerShell の言語罠**: sh の列挙関数が偽値を返し `var=$(func)` で停止（D2）／PowerShell の
  単一要素配列アンラップで `$x[0]` が先頭1文字化（D4・review must）。同種は再発しやすい。
- **③ reworks の記録フィデリティ欠落**: `aidev-worktree` は **sent_backs=1 だが reworks=0**（差し戻し後に
  `coding start` を再記録しなかった）。一方 `sndpgmmsg` は reworks=3 > sent_backs=2。**両者で reworks と
  sent_backs の関係が一貫せず、横断分析（＝本レポートの土台）の信頼性を損なう**。

## systemic な前進（このサイクルで実現したこと）

- **検証の穴を恒久封鎖**: 前回まで CI は無く、検証は手元実行＋人手注記頼みだった。本サイクルで **CI が誕生**し、
  pwsh 依存の ps1 検証（最大の穴）が自動で回るようになった（#29）。skip 可視化（#32）と合わせ、
  「環境不足で未検証のまま緑」を構造的に検出できる。
- **`retro → issue → 実装 → merge` ループの実証**: 前回提案した自己給餌ループ（insights→propose→batch→PR）の
  一部が、本セッションで **retro 起点・対話駆動で一巡**した（5 提案すべてが PR 化・マージ）。propose/batch を
  介さずとも、retro の提案が確実に着地する経路が機能することを示した。
