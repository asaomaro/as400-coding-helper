# 振り返り: ルーラー/制御コード表示の対象拡張子の拡張（#4）

## サマリ

AGENTS.md 指定の 7 拡張子（`rpg, rpgle, clp, dds, dspf, prtf, cmd`）で表示系機能（ルーラー/SOSI）が
有効になるよう `fileScope.ts` の対象拡張子を一元化し拡張。実装は `a071525`（`Closes #4`）で main に着地、
issue #4 はクローズ済み。受け入れ基準（7拡張子で真・大文字含む・対象外は偽・既存維持・両関数が同一定義参照）は満たした。

review で**重要な設計差し戻し（D1）**が1件あり、その指摘により「言語同居」方針を撤回して着地した点が本作業の山場。

## うまくいった点

- **review が表面化させた副作用の捕捉**：spec の「`.dds/.dspf/.prtf`→`rpg-fixed`、`.cmd`→`cl` に言語同居」案に対し、
  review が「languageId が変わることで `.cmd` に CL 診断（`parseClDocument`）が走り、`.dds` 等に RPG 編集
  キーバインド（コメントトグル/タブナビ）が作動する」副作用を指摘（D1）。**着地前に**「表示系のみ拡張子ベース・
  言語登録は最小限（`.rpg` のみ `rpg-fixed` 同居）」へ修正できた。AS400 の表示系拡張という issue の意図を守れた。
- **拡張子定義の一元化**：`TARGET_EXTENSIONS` 定数に集約し `isInScopeDocument`/`isInScopeUri` の二重定義を解消。
  非機能要件（一元管理）を構造で担保。
- **decisions.md が判断経緯を残せた**：spec の前提を覆す決定（D1）を、理由・影響つきで記録できた。

## 課題 / 手戻り

- **spec 段で防げた設計ミス（最大の手戻り源）**：spec は「新拡張子を既存言語へ関連付ける」だけを見て、
  **languageId を変えると下流（診断・編集キーバインド）に何が波及するか**を分析していなかった。
  この観点は spec の「振る舞いの詳細／異常系」に含めるべきで、review まで持ち越したのは上流の弱さ。
  → 影響度の高い手戻り（実装方針そのものの変更：T2 破棄 → T5/T6/T7 で置換）。
- **`metrics.yml` が存在しない**：本作業は工程メトリクスを一切記録していない。結果として手戻り回数・
  差し戻し回数・リードタイムが metrics から導出できず、retro の定量分析が decisions.md/tasks.md の
  痕跡頼みになった。
- **`review.md` が存在しない**：review 指摘（D1 相当の must）が protocol §8 の正規ログ（review.md）に
  残っていない。指摘は decisions.md にしかなく、insights 等の横断再発分析からは見えない。
- **deliver が aidev ゲートを迂回**：実装は `a071525` で直接 main に着地し #4 もクローズ済みだったが、
  aidev の deliver ゲート記録は本セッションで事後追記（`9af1799`）した。状態と実体が一時的に乖離していた。

## 改善提案

### 製品 / コード（→ issue 候補）

- **フォーマット固有のルーラー桁定義の不在**：`.dds/.dspf/.prtf/.cmd` は現状 RPG のルーラー桁前提で表示される
  可能性がある（本 issue では「表示有効範囲の拡大」のみがスコープで、各フォーマット固有の桁位置定義は対象外と明記）。
  DDS/PRTF/CMD は桁意味が RPG と異なるため、**各フォーマット固有のルーラー桁定義**を別 issue 化する候補。
- **新拡張子の回帰テスト観点**：`.dds` 等で「RPG 編集機能が付かないこと／CL 診断が走らないこと」を担保する
  副作用回帰テストは現状ユニットの scope 判定中心。languageId 非付与を保証するテストの追加余地。

### PJ プロセス / 規約（→ AGENTS.md）

- **「languageId / アクティベーションを変える変更は下流波及を spec で評価する」チェックの明文化**：
  VSCode 拡張では languageId が診断・キーバインド・補完の発火条件を兼ねる。拡張子↔言語の関連付けを
  いじる際は「その言語に紐づく既存 contributes（diagnostics/keybindings/snippets）に副作用が出ないか」を
  spec 段で点検する観点を AGENTS.md（または spec の観点）に追補したい。今回の D1 はこの観点があれば spec で潰せた。

### ハーネス自体（→ aidev-* への提案・適用は人間）

- **deliver 記録の徹底（迂回検知）**：実装が aidev ゲート外で main へ着地し issue クローズ済み、という乖離が起きた。
  deliver 工程の冒頭で「対象 issue が既にクローズ／実装が既に着地していないか」を検知し、
  検知時は「事後記録モード」を明示する手順を `aidev-70-deliver` に足すと、状態と実体の乖離に気づきやすい。
- **metrics.yml / review.md の生成を工程の必須出力として強制**：本作業は両方とも欠落していた。
  各工程 skill（特に start 時の metrics 追記、review の review.md 追記）が**ファイル不在時に生成して追記**する
  ことを protocol §8 の「必須」として明記し、retro/insights の定量分析が常に成立するようにする。
  （欠落時に retro でフラグを立てる仕組みでもよい。）
