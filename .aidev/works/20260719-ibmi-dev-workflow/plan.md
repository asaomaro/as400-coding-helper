# 計画: VS Code を軸とした IBM i 開発ワークフローの設計

## 実装方針

成果物は文書 2 点（設計書＋バックログ）。spec で章立てと内容が確定しているので、
分解は「章のまとまり」単位とする。subtask 分割はしない（文書のみ・1 PR で完結する規模）。

書く順序は「根拠が固い部分から」: 資源マップ（research 直結）→ フロー（設計の中核）→
環境・安全規則 → 確認計画 → バックログ。バックログは設計書の全章が書けてから起票する
（章を書く過程で項目の過不足に気付くため）。

## 作業順序と依存関係

1. T1: 設計書の骨格＋1-2 章（全体像・資源マップ）（依存: なし）
2. T2: 3-4 章（人間のフロー・AI のフロー 4 役割）（依存: T1）
3. T3: 5-7 章（環境・安全規則・未検証事項と確認計画）（依存: T1）
4. T4: バックログ起票 `.aidev/backlog/workflow.md`（依存: T2, T3）
5. T5: 設計書 8 章（バックログ参照）＋全体整合（依存: T4）

## リスク / 留意点

- **研究成果との乖離**: 設計書の主張は research.md の F 番号に紐づける。
  根拠の無い主張を書かない（test 工程で突合検査する）
- **バックログ形式**: 既存 `.aidev/backlog/*.md` の frontmatter（backlog/kind/priority）と
  「1 未チェック行 = 1 タスク」の形式に合わせる。`aidev-util-batch` が消化できる形を保つ
- **実装先が他リポジトリの項目**: as400-web-emulator 分は行内に実装先を明記し、
  本 PJ の batch が誤って着手しない書き方にする（`(repo:as400-web-emulator)` 等）
- pub400 が復旧しても、本作業では実機確認を**しない**（P1 タスクとして起票するだけ。
  スコープを文書に保つ）

## テスト方針（test 工程）

文書のみなので、機械的に検査できるものを検査する:

- 設計書の全主張が research.md の F1〜F9 または既存文書へ参照を持つこと（サンプリングで
  なく全主張を目視走査）
- requirement の完了条件 7 項目それぞれに対応する記述が設計書にあること（spec の
  対応表どおりか）
- バックログ: 全項目に優先度と実装先があり、P1 に依存漏れが無いこと。frontmatter が
  既存形式と一致すること
- リンク切れ（相対パス）が無いこと
- 未検証事項（実機確認 2 点）が「確認計画」として明示され、確定事実と混ざっていないこと
