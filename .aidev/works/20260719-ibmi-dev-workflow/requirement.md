# 要件: VS Code を軸とした IBM i 開発ワークフローの設計

## 背景 / 課題

- IBM i（AS400）の開発を SEU/5250 から VS Code 軸へ移す流れがあり、事実上の標準として
  Code for IBM i（＋IBM の拡張パック）が存在する。活発で接続・コンパイル・デプロイの
  基盤は厚い（`docs/research/code-for-ibmi.md` で調査済み）。
- しかし次の不足がある:
  1. **AI エージェントから使えない**。Code for IBM i の機能（コンパイル Ctrl+E、エラー一覧、
     メンバー操作等）は VS Code の UI に閉じており、AI エージェントがワークフローの主体に
     なれない。ここが本作業の中心課題。
  2. **固定長・日本語・DDS の編集支援が薄い**。この不足は本ワークスペース
     （as400-coding-helper）が既に補っている（F4 プロンプター・ルーラー・SOSI・補完）。
  3. **リンターが固定長に効かない**。vscode-rpgle のリンターは `**FREE` 限定。
- 使える資源は 3 つ: **Code for IBM i**（活用する側に回る）、**本ワークスペース**、
  **as400-web-emulator**（https://github.com/asaomaro/as400-web-emulator ）。
  as400-web-emulator は MCP サーバーで 5250 を AI から操作でき（サインオン・画面取得・
  キー送信・スプール取得）、pub400 実機で E2E 検証済み。

## 目的 / ゴール

VS Code を軸とした IBM i 開発ワークフローを**設計**し、そのために**必要な機能を洗い出し**、
実装すべきものを**優先度付きバックログ**に落とす。

基本の形は **ローカル・ファースト**: コードは VS Code などのローカル環境で書き
（人間も書くので本 PJ のルーラー・F4 プロンプターが現役で効く）、書いたコードを
IBM i へ転送してコンパイル・テストする。

達成したい状態:
- 人間の開発フロー（編集→転送→コンパイル→テスト→レビュー）と、AI エージェントの
  開発フロー（同じ流れを AI が自律で回す）の両方が 1 つの設計にまとまっている。
- 「何が既にあり、何が足りず、何を作るか」が根拠付きで整理されている。
- 足りない機能が、後続の作業として個別に着手できる粒度で起票されている。

## スコープ

### 対象

- 3 資源の実地調査（機能・AI からの到達可否・現状の限界）
- **ソースの送受信経路の整理**: ローカル ⇔ IBM i の転送は **IFS ではなく通常の
  ライブラリー上のソースメンバー**（QSYS.LIB のソース物理ファイル）に対して行いたい。
  送り（ローカル→メンバー）だけでなく**受信（メンバー→ローカル）**も対象。
- AI エージェントに担わせる 4 役割の実現方法の整理:
  1. コーディング〜転送〜コンパイル〜修正の自律ループ
  2. テスト:
     - **RPGUnit によるテスト Suite 設計**も視野に入れる（ユニットテスト）
     - **DB2 接続の MCP 等を活用したテストデータ作成**と、AI による自動テスト
     - 5250 アプリの E2E テスト（MCP / Web エミュレーター）
  3. コードレビュー・リンティング（固定長の規約検査を含む）
  4. 運用・調査タスク（スプール確認・ジョブ調査・メッセージ確認等）
- 固定長ソース向けリンターの要件整理
- ワークフロー設計書の作成（導入手順・構成を含む）
- 実装バックログの起票（優先度付き）

### 対象外

- 洗い出した個別機能の実装（それぞれ別の作業として起こす）
- Code for IBM i 自体への貢献（fork・PR）の判断
- チーム展開の実施（設計上の考慮には含めるが、展開作業はしない）

## 機能要件（成果物が満たすべきこと）

- 3 資源それぞれについて「AI エージェントから使えるもの／使えないもの」の一覧が
  根拠（ソース・ドキュメント・実測）付きで示されている
- 4 役割それぞれについて、実現経路（既存資源で足りる／補完が要る）が示されている
- AI が UI に閉じた機能を代替する経路（例: ssh 直・MCP・CLI）が具体化されている
- ソースメンバー送受信（双方向）の経路が、文字コード・ソース日付の扱いを含めて
  判定されている
- リンターについて、固定長で検査すべき項目と既存資産（本 PJ の桁定義・検証スクリプト）の
  再利用可否が整理されている
- ワークフロー全体図（人間フロー・AI フロー・両者の接点）が図示されている
- バックログの各項目に、価値（どの役割・課題に効くか）と優先度が付いている

## 非機能要件 / 制約

- **段階導入**: まず作成者一人で回して有効性を判断し、その後チーム（受託案件）へ展開する。
  一人で完結する構成を先に、チーム前提（共有設定・導入手順・習熟）は設計上の考慮に留める。
- **実機**: 当面 pub400 を検証環境とする。IBM の無料検証用クラウド環境が割り当て可能なら
  それを使いたい（有無・条件は未確認 → 調査項目）。
- **共用機への配慮**: pub400 は無償の共用機であり、大量バッチ・高頻度アクセスを設計に
  組み込まない（AGENTS.md の既存規約に従う）。
- **日本語環境が前提**: DBCS/SOSI・日本語ヘルプの扱いを落とさない。
- **Code for IBM i は活用する**（競合ではなく補完。既存調査の結論を維持）。
- **実機アクセスの許可範囲**: SQL や SSH で接続してシステム情報（ジョブ・オブジェクト・
  メッセージ・スプール等）へアクセスすることは実施してよい。

## 未確定事項 / 確認したいこと

いずれも research で解消する（ユーザーへの質問ではなく調査項目）:

- **IBM の無料検証環境**: 個人/開発者向けに検証用 IBM i 環境の無償提供があるか、
  条件・制約は何か（あれば pub400 の負荷制約から解放される）
- **Code for IBM i の AI 到達性**: 機能のうち何が UI 限定で、何が別経路（内部 API・
  設定ファイル・コマンド）で叩けるか。実装を読んで判定する
- **as400-web-emulator の現到達点**: MCP ツールの一覧と粒度、E2E がどこまで書けるか、
  本ワークフローに足りないものは何か
- **固定長リンターの既存資産**: 固定長 RPG/CL/DDS 向けのリンターが世に存在するか。
  本 PJ の桁定義・往復検証をリンターに転用できるか
- **AI 自律ループの実機経路**: コンパイル・エラー取得を AI が ssh 等で直接行う経路の
  実現性（本 PJ で実績のある ssh + system コマンドの延長で足りるか）
- **ソースメンバー送受信の経路**: ライブラリー上のソースメンバーへの送受信を何で行うか。
  Code for IBM i のメンバーブラウズ／本 PJ 実績の `CPYFRMSTMF`（送りのみ実績あり）／
  その他の経路。受信側・文字コード（CCSID・DBCS）・ソース日付の扱いを含めて確認する
- **RPGUnit の現状**: RPGUnit（および IBM の vscode-ibmi-testing 等のテスト拡張）の
  導入要件・実機（pub400）で使えるか・AI から実行できるか
- **DB2 接続 MCP の既存実装**: Db2 for i に繋ぐ MCP サーバーが既に存在するか
  （テストデータ作成・結果検証に使えるか）。無ければ自作の要否をバックログで判断
- **aidev ハーネスとの関係**: この開発ワークフロー設計と、既にある aidev フロー
  （requirement→…→deliver）の役割分担をどう位置づけるか
