# 要件: レイアウト診断を lint / エディタ診断に届ける

外部チケット: #105

## 背景 / 課題

`dspfLayout` / `prtfLayout` はレイアウトを解決するときに診断を出しているが、
**プレビューを開いた人にしか見えない**。エディタの診断にも `lint` CLI にも出ない。

そのため、実機でコンパイルが通らない記述に気付けない。たとえば:

- **1 桁目に項目を置いた**（原典: 最初の桁は属性文字のために予約。実機でも作成不可）
- **画面／紙面をはみ出した**
- **位置欄が数字でない**

いずれも「プレビューを開けば分かる」が、**プレビューを開く習慣が無ければ素通りする**。
コンパイルして初めて分かるなら、この拡張機能を使っている意味が薄い。

AGENTS.md の「**追加したリソースは『到達可能』になって初めて完了**」がそのまま当てはまる。
診断のロジックは既にあり、**届いていないだけ**。

PR #104（DSPF 画面プレビュー）の retro で検出した。

### 既存 lint との関係

`lint` は既に `.dspf` / `.prtf` を対象にしており（`resolveSourceKind` 経由）、
エディタ診断と CLI の両方に出る経路が通っている。**枠組みは揃っている**。

ただし既存の 4 規則は**プロンプター定義から機械的に導かれる、行単位の検査**で、
レイアウト診断は**ファイル全体を解決してから出る**という違いがある。

## 目的 / ゴール

**レイアウトの誤りを、プレビューを開かなくても気付けるようにする。**

編集中はエディタに出て、CI では `lint` CLI で落とせる。

## スコープ

### 対象

- **DSPF と PRTF の両方**のレイアウト診断を届ける。
  - DSPF（7 種）: `column-one-reserved` / `overflow` / `overlap` /
    `missing-position` / `invalid-position` / `relative-position-unresolved` /
    `invalid-screen-size`
  - PRTF: `possible-overprint` / `spacing-with-line-number` /
    `spacing-with-conflicting-keyword` / `overlap` / `overflow` /
    `out-of-range` / `invalid-position`
- **エディタ診断と `lint` CLI の両方**に出す（SARIF にも乗る）。
- **診断ごとに既定 ON / OFF を決める**。判断基準は既存方針を踏襲する。
- 利用者が設定で診断ごとに切り替えられること（既存の `rpgClSupport.lint.rules` と同じ形）。

### 対象外

- **診断そのものの精度向上**。`overlap` の条件付け同値判定や `+n` の解決は別課題
  （retro の提案 4・issue 候補 1）。**今あるものを届けることに徹する**。
- **新しい診断の追加**。既存のものを届けるだけ。
- **プレビュー側の表示の変更**。プレビューは今のまま。
- 実機との通信。

## 機能要件

- `.dspf` / `.prtf` を編集すると、レイアウト診断が**エディタの問題パネルに出る**。
- `lint` CLI（`npm run lint`）でも同じ診断が出て、**CI で落とせる**。
- 診断は**ソースの行**を指す（`sourceLine`）。可能なら桁も指す。
- 既存 4 規則と**同じ設定の形**で診断ごとに ON/OFF を切り替えられる。
- **既定 ON にするのは、実機コンパイル確認済みのサンプルに当てて偽陽性 0 のものだけ**。
  0 でないものは既定 OFF とし、その理由を型のコメントに残す（既存 `RuleId` と同じ流儀）。

## 非機能要件 / 制約

- **`lint` core は vscode 非依存を保つ**。`verify-lint-core.mjs` の純粋性検査を通す。
- **同じロジックがエディタと CLI で動く**こと（既存の設計方針。「エディタでは出ないが
  CI で落ちる」食い違いを作らない）。
- **プレビューとレイアウト解決を二重に持たない**。診断の出どころは
  `dspfLayout` / `prtfLayout` の 1 か所のまま。
- **既存の 4 規則・プレビュー・書き戻しに退行がない**。
- 日本語のメッセージ（既存の診断メッセージをそのまま活かす）。

## 完了条件 (受け入れ基準)

- [ ] `.dspf` の 1 桁目に項目を置くと、**エディタの問題パネル**に指摘が出る
- [ ] 同じ指摘が `npm run lint` の出力にも出る（SARIF にも含まれる）
- [ ] `.prtf` のレイアウト診断も同様に出る
- [ ] **実機コンパイル確認済みのサンプル**（`CUSTMNT.dspf` / `CUSTRPT.prtf` ほか
      `docs/src/` の DDS 全部）に当てて、**既定 ON の診断で偽陽性 0 件**
- [ ] 偽陽性が出た診断は**既定 OFF** にし、理由が型のコメントに残っている
- [ ] 診断ごとに設定で ON/OFF を切り替えられる
- [ ] `verify-lint-core.mjs` が通る（`lint` core の純粋性が保たれている）
- [ ] 既存の 4 規則の挙動が変わっていない
- [ ] プレビューの表示・書き戻しに退行がない

## 未確定事項 / 確認したいこと

いずれも **research / spec で解消する**。

- **既存 `Rule` の枠組みに乗るか**。既存規則は行単位（`positional` の別があり、
  前処理が「欄が存在する行」を判定してから回す）。レイアウト診断は
  **ファイル全体を解決してから出る**ので、`Rule` の形が合わない可能性がある。
  枠組みを広げるのか、別の経路にするのかを決める必要がある。
- **`RuleId` の粒度**。診断コード 1 つ＝規則 1 つにするのか、
  「レイアウト」でひとまとめにするのか。設定の切り替え粒度に直結する。
  PRTF と DSPF で同名の診断（`overlap` / `overflow` / `invalid-position`）があり、
  ID の衝突をどう扱うかも決める。
- **偽陽性の実測値が未知**。`docs/src/` の DDS サンプルに当てた結果を見ないと
  既定 ON/OFF が決められない。**先に実測する**（research の主目的）。
  特に PRTF の `possible-overprint` は「行番号もスキップもスペースも無い」で出るため、
  ノイズが多い可能性が高い。
- **診断の桁情報**。既存 `LintFinding` は `startColumn` / `endColumn` を持つが、
  レイアウト診断は `sourceLine` しか持たない。桁を出すなら診断側に足すことになり、
  「対象外: 診断そのものの変更」との線引きが要る。
- **重複の扱い**。プレビューを開いた状態で編集すると、同じ問題が
  プレビュー内とエディタの両方に出る。これは許容でよいか。
