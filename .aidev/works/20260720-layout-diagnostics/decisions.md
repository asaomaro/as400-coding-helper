# 決定記録

## D1: 既存テスト 2 件の期待値を更新した（意図した変更）

- 背景: `RuleSpec` に `kind` の判別子を足し、既定 ON の規則を 3 → 7 に増やしたため、
  既存の `lintRules.test.ts` が 2 件落ちた。
  - `既定で有効なのは 3 規則`（既定の集合を固定していた）
  - `定位置の欄を見ない規則は行長だけ`（`spec.positional` を判別子なしで参照）
- 決定: **どちらも期待値を更新する**。前者は「7 規則（行単位 3 ＋ レイアウト 4）」に、
  後者は `kind === "line"` で絞ってから判定する形に直した。
- 理由 / 代替案: 既定 ON を増やすのは spec で決めた意図した変更で、テストが
  それを検出したのは**正しい振る舞い**。テストを緩める（件数を見ない等）選択もあるが、
  「既定に何が入っているか」を固定する価値のあるテストなので、値を更新して残す。
- 影響: 併せてレイアウト規則向けの検査を 2 件足した
  （ファイル単位の規則の一覧・既定 OFF の 2 件）。

## D2: `missing-position` を初版で lint に流さない

- 背景: spec の「採らなかった診断」に含めた判断の補足。実装中に再確認した。
- 決定: `missing-position`（DSPF）は `RuleId` を割り当てない。
- 理由 / 代替案: 位置欄が空の項目を検出するもので原典上は安全側に見えるが、
  **潜在／メッセージ／プログラム間フィールドの除外漏れがあると大量に誤報する**。
  実際、種別の振り分けを誤ったときに出たのがこの診断で、`.pf` / `.lf` の
  全フィールドに出た（実測 15 件）。母数 2 では除外の網羅性を検証できない。
- 影響: 受け入れ基準には含まれない（spec で対象外と明記済み）。
  サンプルが増えたら `RuleId` を足すだけで有効化できる。
