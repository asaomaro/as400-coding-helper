## うまくいった点

### 原典の直読が 2 回、既定の理解を覆した

AGENTS.md の「原典照合は主エージェントが生テキストを直読する」が、そのまま効いた。

**research（訂正 3 件）**:
- 位置欄に **`+n` 相対桁**がある（原典 `DSPSIZ` 例 4 の「DDS のプラス機能」）。
  「DSPF にカーソルは不要」という当初の整理は不正確だった。
- 条件付け欄には**画面サイズ条件名**も入る。標識だけ読むと 2 次画面用の位置指定を落とす。
- `*DSw` の展開表は **`CRTDSPF` から取れない**。CDML を grep して
  `DSPSIZ` パラメータが**存在しない**ことを機械的に確認した（DDS キーワード専用）。

**review（must 1 件）**:
- 数値形式の `DSPSIZ` で `*DS3` 条件の項目が黙って消える欠陥を、
  原典の「ユーザー定義名を指定しない場合には IBM 提供名を使用して条件付ける」から特定した。

いずれも**知識で答えていたら間違えていた**。特に 3 点目は「CL コマンドのパラメータから既定値を
取る」という前作業（PRTF の `CRTPRTF`）の成功体験がそのまま誤った類推を生んでおり、
実データに当たらなければ気付けなかった。

### 「描けないものは描かず、理由を示す」を spec で方針として立てた

DSPF は PRTF より未解決要素が多い（参照・`+n`・キーボードシフト）。
spec の方針 2 としてこれを明文化したことで、実装中の判断（`+n` をどうするか、
幅不明をどう描くか）が毎回ぶれずに決まった。**review の must もこの方針違反として
検出できた**（消えていること自体が方針に反する）。

### 共有部品の切り出しを第 1 段に置いた

振る舞い不変なので**既存テストが緑のままであることが即座に正しさの証明**になり、
かつ DSPF 側が「借りる先」を確定させた。二重実装の芽を先に潰せている。
PRTF への退行は実サンプル全行の恒等性テストが担保した。

## 課題 / 手戻り

### ① research で判明した事実が、実装で 1 段落ちた（review の must）

**これが最も重い学び。**

research は「条件付け欄には画面サイズ条件名も入る」を正しく突き止め、
spec にも「画面サイズ条件名は別軸で扱う」と書いた。
しかし**どう突き合わせるか**（名前か、サイズか）まで書かなかったため、
実装は素直に名前の文字列比較になり、数値形式の `DSPSIZ` で項目が消えた。

**前段で防げたか**: 防げた。spec の「振る舞いの詳細 4.」に
「条件名が 1 次の条件名と一致する場合のみ配置」と書いており、
**この記述自体が誤り**だった（原典は名前が無くても IBM 提供名で条件付けられると言っている）。
research の生テキストには根拠があったのに、spec を書くときに読み返さず要約から書いた。

### ② テストが誤った期待値を固定していた（review の should）

はみ出し判定のテストは「76 桁目・幅 5 で overflow」を期待しており、**通っていた**。
だが原典の「文字フィールドの最大桁数は画面サイズから 1 を引いた桁数」と突き合わせると、
これは**正当な配置**で誤検出だった。テスト自体がバグだった。

**前段で防げたか**: 防げた。AGENTS.md には
「一度出た欠陥はテストに足す」「テストを足したら直す前に落ちることを確かめる」はあるが、
**期待値そのものを原典で裏取りする規約が無い**。
「落ちることを確認する」は*テストが機能すること*は保証するが、*期待値が正しいこと*は保証しない。

### ③ 工程イベントの記録漏れ 3 か所（私の手順ミス）

`metrics.yml` を読み返して判明した。**捏造せず事実として残す**。

| 漏れ | 影響 |
|---|---|
| `research` の差し戻し後に `start` を再記録していない | 手戻り回数に計上されない（start 1 回のまま） |
| `review` の `start` を**一度も記録していない**（両ラウンドとも） | review の所要時間が導出できない |
| 差し戻し後の `coding` に `approved` が無く、`test` の 2 巡目も未記録 | 再 coding・再 test の所要が測れない |

いずれも `aidev guard <工程>` は実行したが `aidev event <工程> start` を忘れたもの。
**guard と event start が別コマンドで、guard だけでも工程に入れてしまう**のが誘因。

### ④ `out-test/` の残骸で偽のテスト件数が出た（decisions D3）

改名直後にテストが 338 → **348** に増えた。旧名のコンパイル結果が残って二重実行されていた。
`rm -rf out out-test` で 338 に戻ることを確認したが、**一時的に「テストが増えた」と誤認**した。
逆方向（消したはずのテストが通り続ける）だと**気付けない**危険がある。

## 改善提案

### 製品 / コード（→ issue 候補）

1. **`+n` 相対桁の解決**（初版で未対応）。原典に折り返し規則の明示が無いため、
   実機で `CRTDSPF OPTION(*SRC)` の拡張ソース印刷出力を見て確定する。
   `ibmi-remote` skill が使える。副作用の無い形（コンパイルして出力を見るだけ）で確認できる。
2. **キーボード・シフト（35 桁目）による表示桁数**の対応表を原典から取得し、
   幅の解決に反映する。現在はプログラム桁数で描いており**実機より狭く出ることがある**。
3. **2 次画面サイズの描き分け**。現在は 1 次のみ描画し注記に出している。
   切り替え UI を持つと WebView に状態を持つことになるため、設計判断が要る。
4. **重なり判定の精度向上**。現在は「両方とも無条件のときだけ報告」と保守的。
   標識の同値判定（`01` と `N01` の背反など）に踏み込むかは要検討。
5. **`.mnudds` の実物サンプルを `docs/src/` に追加**。現在リポジトリに存在せず、
   合成ドキュメントでしか検証できていない。実機でコンパイルが通るものを 1 本置きたい。
6. **レイアウト診断を lint / エディタ診断に届ける**（PRTF・DSPF 共通の既存の穴）。
   `dspfLayout` の 7 種（`overlap` / `overflow` / `column-one-reserved` /
   `missing-position` / `relative-position-unresolved` / `invalid-position` /
   `invalid-screen-size`）と `prtfLayout` の診断は、**プレビューを開いた人にしか見えない**。
   「1 桁目に置いた」「画面をはみ出した」がエディタにも `lint` CLI にも出ない。
   AGENTS.md「追加したリソースは到達可能になって初めて完了」がそのまま当てはまる。

   **ただし単純に既定 ON にしてはいけない**。`lint/rules/index.ts` の方針:
   > 既定 ON にしてよいのは「原典で裏が取れ、実機コンパイル確認済みのソースに
   > 当てて偽陽性が 0 件だったもの」だけ。4 規則をそのまま当てると検証済みの
   > ソースに 30 件の指摘が出ることを実測してある。

   レイアウト診断も同じ検証を通す必要がある。特に `overlap` は条件付けの
   同値判定に踏み込んでおらず**保守的に倒したばかり**（提案 4 と関係する）。
   実サンプル群に当てて偽陽性 0 を実測してから既定を決める。

### PJ プロセス / 規約（→ AGENTS.md）

1. **「テストの期待値も原典で裏を取る」を明記する**（課題②）。
   既存の「テストを足したら直す前に落ちることを確かめる」の直後に置くのが適切。

   > **落ちることの確認は、期待値の正しさを保証しない。** 原典に規定のある値
   > （桁・上限・境界）をテストに書くときは、**期待値そのものを原典の生テキストで裏取りする**。
   > 実際、DSPF のはみ出し判定で「76 桁目・幅 5 で overflow」という**誤った期待値が
   > テストとして固定され、通っていた**。原典の「最大桁数は画面サイズから 1 を引いた桁数」と
   > 突き合わせて初めて誤りと分かった。

2. **「原典から得た事実は、要約でなく生テキストに戻って spec に落とす」**（課題①）。

   > research の要約から spec を書くと、**判断に効く条件が落ちる**。
   > 実際、「条件付け欄に画面サイズ条件名が入る」までは spec に書けたが、
   > 「名前が無くても IBM 提供名で条件付けできる」が落ち、実装が名前比較になって
   > 項目が黙って消えた。spec に規則を書くときは research.md の引用元まで戻る。

3. **`out-test/` の残骸に関する注意を「テストの走らせ方」節に追記**（課題④）。

   > `npm test` は `out-test/` をクリーンしない。**改名・削除を伴う変更の直後は
   > `rm -rf out out-test` してから走らせる**。旧名の成果物が残ると二重実行され、
   > 件数が水増しされる（実際に 338 → 348 になった）。逆に、消したはずのテストが
   > 通り続ける形だと気付けない。

   あるいは **`npm test` に `rimraf out-test` 相当を前置する**（コード側の対処）。
   こちらの方が確実なので、issue 候補にも挙げる。

4. **競合との差分は「桁がずれる理由」を数えて書く**（うまくいった点の一般化）。
   IBM i Renderer は DSPF で **DBCS と属性文字の二重にずれる**。
   この種の「なぜ我々が要るか」は README・PR に定量で書けると強い。

### ハーネス自体（→ aidev-* への提案・適用は人間）

1. **`aidev guard <工程>` が成功したら `start` イベントも記録する**（課題③）。

   現在 `guard` と `event start` は別コマンドで、**guard だけでも工程に入れてしまう**。
   実際この作業で 3 か所の記録漏れが出た（review の start は両ラウンドとも欠落）。
   guard は「工程に入ってよいか」の判定なので、**通過＝工程開始**とみなすのが自然。
   別々に保ちたいなら、`guard` の出力に「`aidev event <工程> start` を忘れずに」と
   促す 1 行を足すだけでも効く。

2. **`aidev verify` に「approved があるのに start が無い工程」の検査を足す**（課題③）。

   記録漏れは**後から気付けない**（今回は retro で metrics を読み返して初めて分かった）。
   verify は既に不変条件を見ているので、ここに「イベントの対」の検査を足せば
   deliver 前に検出できる。同様に「sent_back の後に同一工程の start が無い」も検出できる。

3. **差し戻し後の再開手順を、skill 側でチェックリスト化する**（課題③）。

   protocol.md「3.」には「再開する時に `aidev event <工程> start` を記録する」と
   書いてあるが、**差し戻しを実行する時点では次の行動が先にある**（指摘の修正）ため、
   記録が後回しになって落ちる。review / test skill の差し戻し分岐に
   「① sent_back を記録 → ② 差し戻し先の start を記録 → ③ 修正に入る」と
   順序を明示すると落ちにくい。

4. **research → spec の引き継ぎに「申し送りの消費確認」を入れる**（課題①）。

   research.md には「spec への申し送り」節があり、今回も
   「条件付け欄は 2 形態」と正しく書かれていた。だが**spec がそれをどう消費したかを
   突き合わせる手順が無い**。spec skill に「research.md の申し送り各項目が
   spec のどこに落ちたかを対応表で示す」を足すと、今回の落ちが防げた可能性が高い。
   AGENTS.md の「追加したリソースは到達可能になって初めて完了」と同じ発想を、
   工程間の申し送りにも適用する形。

## この作業から一般化できること

**「調べた」と「仕様に落ちた」と「実装に効いた」は別物**である。

今回の must は、research では**正しく調べられており**、spec にも**言及があった**のに、
実装では**効いていなかった**。AGENTS.md が既に持っている
「追加したリソースは到達可能になって初めて完了」「プロンプターはモデルまででは届いていない」
という教訓は、**コードだけでなく工程間の情報の受け渡しにも当てはまる**。

research の 67 分は無駄ではなく、むしろ**その 67 分の成果を spec と実装に
取りこぼさず届ける仕組み**が、次に投資すべきところだと考える。

  変更は即座に**全プロジェクトへ影響**し、履歴も差分レビューも残らない。
  この状態自体が次の改善候補（ハーネスのバージョン管理）。
