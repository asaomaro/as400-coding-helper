# 決定記録

## D1: 移設した関数のテストは移さず、新規に単体テストを足す

- 背景: plan の T3 / T4 は「`prtfLayout.test.ts` の D4 回帰テストと幅の解決 suite を
  移設先へ移す」としていた。しかし既存テストはいずれも `resolvePrtfLayout` を
  経由して検証しており、関数を直接呼んでいない（＝結合テストの形）。
- 決定: **既存テストは `prtfLayout.test.ts` に残す**。移設した共有モジュールには
  **新規に直接の単体テストを足す**。
- 理由 / 代替案: 既存テストは「PRTF の振る舞いが変わっていないこと」の担保として
  価値がある。移設すると PRTF 経路の回帰網が薄くなり、今回の切り出しで
  最も避けたいこと（R7: PRTF への退行）に対する守りが弱くなる。
  テストを書き換えると「切り出しが正しいこと」と「テストが正しいこと」を
  同時に疑うことになり、失敗時の切り分けが難しくなる。
- 影響: tasks.md の T3 / T4 の記述と実態がずれる（該当箇所を実態に合わせて更新した）。
  共有モジュールの単体テストは DSPF 側の作業（T5 以降）と同じ回で足す。

## D2: `toLogicalUnits` に「条件付けの前置き行」の判別を足した

- 背景: 原典（`条件付け (7 - 16 桁目)`）より、条件が複数行に分かれる場合
  **項目は最後の標識の行に書く**。つまり先行する行が条件の続きで、
  キーワード継続行が「直前の単位に付く」のと**向きが逆**になる。
  元の `toLogicalUnits` は前者を後者として扱い、直前の単位に連結していた。
- 決定: **キーワード欄が空で条件付け欄に何か書かれている行**は、
  次の単位への前置き（`conditioningLines`）として持ち越す。
- 理由 / 代替案: DSPF の条件付けを読むには先行行が要る。DSPF 側だけで
  別の走査を書くと、論理単位の切り方が 2 つになり食い違う余地ができる。
- 影響: **PRTF の振る舞いは変わらない**。キーワード欄が空の行を直前に連結しても
  空文字を足すだけで `keywords` が変化しないため（`npm test` 338 件で確認済み）。
  `LogicalUnit` に `conditioningLines` を追加したが、PRTF は参照しない。

## D3: `out-test/` の残骸で偽のテスト件数が出る（作業上の注意）

- 背景: T1 / T2 でファイルを改名した際、テスト件数が 338 → 348 に増えた。
  原因は `out-test/` に**旧名のコンパイル結果が残り、二重に実行**されていたこと。
- 決定: 改名を伴う変更では **`rm -rf out out-test` してから `npm test`** する。
- 理由: `npm test` は `out-test/` をクリーンしないため、改名・削除の直後は
  古い成果物が実行され続ける。「テストが増えた／消えたはずのテストが通る」という
  紛らわしい結果になる。
- 影響: 検証手順のみ。コードには影響しない。CHECKLIST か AGENTS.md への
  記載を retro で検討する。
