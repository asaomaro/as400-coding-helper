### [must] `dspfLayout.ts` 画面サイズ条件名を**名前の文字列だけ**で突き合わせている — 項目が黙って消える

`resolveDspfLayout` は 1 次画面サイズの条件名と項目の条件名を
文字列比較しており、`DSPSIZ` が**数値形式で条件名を持たない**場合に
`primaryCondition` が `undefined` になる。その状態で `*DS3` に条件付けた項目は
一致せず、**配置もされず診断も出ない＝画面から黙って消える**。

原典（`表示装置ファイルの DSPSIZ`）は数値形式と IBM 提供名の併用を明示している:

> **ユーザー定義の画面サイズ条件名を指定しない場合には、IBM 提供の画面サイズ条件名を
> 使用してフィールドの位置を条件付ける必要があります。**

原典の例 3 がまさにこの形（`DSPSIZ(24 80 27 132)` の後に `*DS4 26 2`）。

実際に動かして確認した:

```
DSPSIZ(24 80 27 132)
R RECORDA
  *DS3  FIELD1  10 0  5  2      ← *DS3 は 1 次(24x80)と同じサイズ
→ items [] / diagnostics []      ← 消えている
```

spec の方針 2「描けないものは描かず、**理由を示す**」に反する
（理由も示さず消えており、利用者は項目を書き忘れたと誤解する）。

**対応**: 条件名の比較を**名前ではなくサイズで**行う。`*DS3` / `*DS4` は
既知のサイズに解決してから 1 次画面サイズと比較し、ユーザー定義名のときだけ
名前で突き合わせる。`isScreenSizeConditionName` の隣に解決関数を置くのが素直。

**追加すべきテスト**: 数値形式 `DSPSIZ(24 80 27 132)` ＋ `*DS3` 条件の項目が
配置されること。`*DS4` 条件の項目は配置されないこと。

### [should] `dspfLayout.ts` はみ出し判定が終了属性文字を含むため、**原典が認める最大幅を誤検出**する

`overflow` の判定に `occupancy.end > screen.columns` を使っている。
`occupancy.end` は終了属性文字の桁なので、**データが画面内に収まっていても**
終了属性文字が画面端の 1 つ外に来ると、はみ出しとして報告される。

原典（`桁数 (30 - 34 桁目)`）:

> **文字フィールドの最大桁数は、表示画面サイズから 1 を引いた桁数**です
> （この 1 桁は**開始**属性文字のためのスペースです）。

80 桁画面での最大は幅 79。開始属性文字が 1 桁目、データが 2-80 桁目となり、
**終了属性文字の場所は無い**。原典はこれを最大値として認めているので、
終了属性文字が画面に載らないことは異常ではない。

実際に動かして確認した（いずれもデータは画面内）:

```
幅79 2桁目  → データ終端 80  診断:[overflow]   ← 原典が認める最大幅
幅5 76桁目  → データ終端 80  診断:[overflow]   ← 最終桁ちょうど
幅5 77桁目  → データ終端 81  診断:[overflow]   ← これだけが本当のはみ出し
```

受け入れ基準「条件付けが違う項目どうしの重なりを誤検出しない」と同じ
「誤検出しない」の要請が、はみ出しにも当てはまる。

**対応**: はみ出しは**データの終端**（`column + width - 1`）で判定する。
`occupancy` は重なり判定にのみ使う（隣接項目が属性文字を共有できるのは
項目どうしの関係であり、画面端との関係ではない）。

**注意**: 既存テスト `"画面をはみ出す項目を検出する（属性文字を含めて判定）"` は
**誤った期待値を固定している**（76 桁目・幅 5 で overflow を期待）。
テストも直す必要がある。直す前に落ちることを確認すること。

### 指摘に至らなかった確認事項（記録）

点検して**問題なし**と判断したもの。

- **属性文字の占有** `[column-1, column+width]` と端点一致を許す重なり判定は
  原典どおり（実サンプルで occupancy `[24,35]` / `[1,52]` を確認）。
- **`.length` による桁計算の混入なし**（R3）。幅は `printWidth` 経由に一本化されている。
- **`DDS_COLUMNS` の導出に触れていない**（R1）。`contributesSideEffects.test.ts` 通過。
- **PRTF への退行なし**（R7）。実サンプル全行の恒等性テスト通過、390 passing。
- **対象判定が `resolveDdsType` 経由**で拡張子を二重に列挙していない。
- **言語登録・`activationEvents`・設定項目を増やしていない**（PJ 方針どおり）。
- **レコード・レベルの条件付けを見ていない**が、重なりは「両方とも無条件」の
  ときだけ報告するため**偽陽性は生じない**（見落としは仕様どおり保守的）。
- **`decisions.md` D1〜D3** はいずれも妥当。特に D2 は PRTF の挙動を変えないことを
  確認済みで、記録の粒度も適切。

## ラウンド 2（2026-07-20）

ラウンド 1 の must 1 件 / should 1 件を修正し、再点検した。

**判定: 指摘なし → deliver へ**

### 対応内容

**[must] 画面サイズ条件名の突き合わせ** → 修正済み

`dspfScreenSize.ts` に `screenSizeOfConditionName` と `matchesScreenSize` を足し、
**IBM 提供名（`*DS3` / `*DS4`）はサイズに解決してから**突き合わせるようにした。
ユーザー定義名のときだけ名前で比べる。`dspfLayout.ts` の判定をこれに差し替え、
`primaryCondition`（文字列比較のための変数）は削除した。

**[should] はみ出しの判定** → 修正済み

`dataEnd(column, width)`（= `column + width - 1`）を足し、はみ出しは
**データの終端**で判定するようにした。`occupancy` は重なりの判定にのみ使う。
両者の使い分けを doc コメントに原典の引用付きで書いた。

### 追加・修正したテスト

**直す前に落ちることを確認済み**（2 件とも失敗 → 修正後に通過）。

- `DSPF: 画面サイズ条件名は名前ではなくサイズで突き合わせる`（新規 4 件）
  - 条件名を持たない `DSPSIZ(24 80 27 132)` でも `*DS3` の項目を描く
  - `*DS4` の項目は描かない（診断も出さない）
  - 1 次が `*DS4` のときは逆になる
  - ユーザー定義名は名前で突き合わせる
- `はみ出しはデータの終端で判定する`（誤った期待値を固定していたテストを差し替え）
  - 幅 79・2 桁目（原典が認める最大）→ 診断なし
  - 幅 5・76 桁目（最終桁ちょうど）→ 診断なし
  - 幅 5・77 桁目（データが 81 桁目）→ overflow
- `行が画面の外なら検出する`（行方向のはみ出しの担保が消えないよう分離）
