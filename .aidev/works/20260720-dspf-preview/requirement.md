# 要件: DSPF 画面の視覚編集 — DBCS 対応

## 背景 / 課題

PRTF（帳票）の視覚編集を先に作った（`20260720-dds-preview` / PR #103）。
**同じ問題が画面（DSPF）にもある**。DDS で書いた画面は、実行して初めて
レイアウトが分かる。SDA の後継が VS Code に無い。

### 競合は DSPF も対象だが、日本語で破綻するのは同じ

IBM i Development Extension Pack 同梱の **IBM i Renderer**
（`halcyontechltd.vscode-displayfile` v0.1.3）は **DSPF が主対象**（PRTF より力が入っている）。
だが配布 VSIX を展開して直読したとおり:

- **DBCS / SOSI の処理が 0 件**、桁の計算は `line.substring()` と `value.length`
- **書き戻しは無し**（閲覧専用）
- README に「開発中・本番非対応」と明記

サンプル `docs/src/CUSTMNT.dspf` には `'顧客保守'`（実機 10 桁 / JS 4 文字）と
`'顧客番号'` があり、**同じ 6 桁のずれ**が起きる。

### PRTF の資産がほぼそのまま使える

前作業で作ったものは DSPF でも効く。

| 資産 | DSPF での有効性 |
|---|---|
| `core/dbcs.ts` の `printWidth` | **そのまま**（SO + 全角×2 + SI） |
| `core/ddsLayout.ts` の桁・レベル判定 | **そのまま**（A 仕様書は共通） |
| 位置欄の書き戻し（`prtfWriteBack`） | **そのまま**（39-44 桁は DSPF も同じ意味） |
| HTML 生成の方針（桁は計算・箱を権威） | **そのまま** |
| WebView の殻（ソースを唯一の真実にする） | **そのまま** |
| `core/dds/editCode.ts` | DSPF にも `EDTCDE` はある |
| **`prtfLayout` の印刷カーソル** | **不要**（下記） |

## 目的 / ゴール

**画面のレイアウトを VS Code 上で見ながら組めるようにする。**
日本語を含む画面でも、画面に見えるものが実機の 5250 表示と一致する。

閲覧だけでなく、項目を動かしてソースに反映できるところまで（SDA 相当）。

## スコープ

### 対象

- **DSPF（表示装置ファイル）** の視覚編集。`.dspf` と `.mnudds`。
- **画面イメージのプレビュー**: A 仕様書を読み、項目・定数を画面上の行桁に描く。
- **DBCS を実機と同じ桁で扱う**（PRTF と共通の `printWidth`）。
- **属性文字を桁として数える**（下記・DSPF 固有の核心）。
- **画面サイズを `DSPSIZ` から決める**（DSPF は DDS に書かれている）。
- **条件付け（8-16 桁の標識）を読み、重なりの判定に反映する**。
- **書き戻し**: 項目を動かすと位置欄（39-44 桁）に反映する。
- 既存の PRTF プレビューと**同じ操作感**にする。

### 対象外

- **サブファイル（`SFL` / `SFLCTL`）の展開** — 明細行の繰り返し表示は
  実行時のデータ件数に依存する。**1 レコード様式として描く**に留める。
- **ウィンドウ（`WINDOW`）の入れ子表示** — 別レイヤーの描画が要る。初版では扱わない。
- **表示属性の視覚化**（`DSPATR(HI/RI/UL)` / `COLOR`）— 桁の正しさが目的で、
  色や強調の再現は狙わない。ただし**桁を消費するもの**は数える。
- **標識の値を変えて条件付けを切り替える機能** — 静的に「どの条件が付いているか」を
  示すまで。競合の `changeInd` に相当する動的な切り替えは初版では扱わない。
- **`REF` の解決** — PRTF と同じく解決しない（幅不明として示す）。
- 実機との通信。
- PRTF の実装との共通化リファクタリング**そのもの**が目的ではない
  （結果として共有するのは構わない）。

## 機能要件

- `.dspf` / `.mnudds` を開いた状態から、**コマンドで**画面プレビューを開ける。
- **画面サイズを `DSPSIZ` から決める**。`*DS3`(24×80) / `*DS4`(27×132) を読む。
  指定が無ければ既定（24×80）。
- 項目・定数を位置欄（39-41 桁＝行 / 42-44 桁＝桁）どおりに描く。
- **属性文字を桁として数える**。原典より:
  > 表示される各フィールドについて、画面上でのフィールドの表示属性を定義するための
  > **属性文字が 1 つ必要**です。…フィールドの終了属性文字は次のフィールドの
  > 開始属性文字に重ねることができ、したがって、**フィールドとフィールドの間に
  > 必要なスペースは 1 文字分だけ**です。
- **1 桁目に項目を置けないことを検出する**。原典より:
  > フィールドは、表示画面の最初の桁を占めることはできません。
  > **最初の桁は属性文字のために予約されています。**
- **条件付けを読む**。同じ位置に複数の項目があっても、**条件が違えば正当**
  （原典: 相互にオーバーラップするフィールドのうち一時点で表示されるのは 1 つ）。
- 画面をはみ出す項目を検出する。
- 項目を動かすと**位置欄だけ**を書き換える（他の桁に触れない）。
- ソースの行と描画の項目が相互にたどれる。
- **既存の IBM i Renderer と同時に導入しても衝突しない**（拡張子判定・別コマンド）。
- **PRTF プレビューと操作を揃える**（同じ操作で同じことが起きる）。

## 非機能要件 / 制約

- **PRTF で作った資産を再利用する**。同じ計算を 2 つ持たない（AGENTS.md）。
- **桁の真実源は既存の DDS 桁定義**。ここで桁を書き起こさない。
- **`printWidth` を共有する**。SOSI 表示・PRTF と同じ規約にする。
- **ランタイム依存を増やさない**（`dependencies: {}` を維持）。
- 日本語 UI。
- 既存機能（ルーラー・F4・SOSI・補完・lint・アウトライン・PRTF プレビュー）に退行がない。

## 未確定事項 / 確認したいこと

いずれも **research 工程で解消すべき**もの。

- **属性文字の占有をどう数えるか**が最大の論点。原典は
  「開始属性文字が 1 つ必要」「終了属性文字は次の開始属性文字に重ねられる」と書くが、
  **描画上どう表現するか**（項目の左に 1 桁空ける／占有範囲を幅＋1 にする）を
  原典を読んで確定する必要がある。**これを誤ると全項目が 1 桁ずれる。**
- **表示桁数がプログラム桁数と違う場合**がある。原典:
  > 表示桁数は、プログラム桁数と同じかまたはそれより大きくなります。
  > フィールドの表示桁数は、**キーボード・シフト（35 桁目）**のほか、
  > 小数点以下の桁数や編集機能などによって決まります。

  35 桁目（データ・タイプ / キーボード・シフト）が幅に効く。PRTF に無かった要素。
- **`DSPSIZ` の書式**（`*DS3` / `*DS4` / 数値指定 / 条件名）の読み方を確定する。
  原典は「位置の妥当性は DSPSIZ キーワードおよび**画面サイズの条件名**によって決まる」
  と書いており、複数サイズを持てる可能性がある。
- **条件付け（8-16 桁）の書式**と、重なり判定への反映のしかた。
  標識の AND/OR・否定（`N`）をどこまで読むか。
- **PRTF の実装をどこまで共有するか**。`prtfLayout` は印刷カーソルを持つが
  DSPF には不要（位置は常に明示）。共通部分の切り出し方を決める。
- **`.mnudds` を同じ扱いにしてよいか**。既存の `resolveDdsType` は
  `.mnudds` を `DDS-DSPF` に寄せているが、メニュー固有の制約がないか確認する。
