# 振り返り: aidev-batch による CL定義4件（DLTF/RTVJOBA/SNDRCVF/WRKSPLF）

## サマリ

`.aidev/backlog/cl-backlog.md` の未処理CL定義4件を **aidev-batch（autonomous）** で消化。各定義の実処理は
PJ skill `cl-command-def` に委譲（IBM原典を Playwright 取得 → types.ts 準拠でマッピング）。4件すべて
test硬ゲート（JSON＋スキーマ検証）を通過し、**1ブランチ1PR（#17）にまとめてレビュー後マージ**。
処理 4/4 成功・失敗0。レビューで確定的 must 欠陥は0（後述の通り、委譲レビューの must 指摘は全て誤検出）。

## うまくいった点

- **並列subagent × `cl-command-def` 委譲**で4件を短時間生成。主エージェントの context を圧迫せず、
  各定義が IBM原典準拠・types.ts 準拠で揃った。
- **Playwright 取得の道具立てを先に確定**（前回 sndpgmmsg retro の教訓）。research段の取得失敗による
  手戻りはゼロ。CJS interop の1点だけ即修正で済んだ。
- **原典の生テキストを主エージェントで機械照合**して生成物の正確性を確定。dropdown/group の振り分け・
  required・パラメータ集合がいずれも原典一致であることを示せた。
- dropdown/group/text の振り分けが規約どおり（固定値→dropdown／修飾名・ELEM→group／混在→text+help）。

## 課題 / 手戻り

- **委譲レビューのツール権限劣化（最重要）**: 検証を4つのsubagentに委譲したが、各エージェントで
  **Bash/WebFetch が拒否され IBM原典に照合できなかった**。結果、知識ベースで [must] を量産
  （DLTF「SYSTEM既定=*FILETYPE」、WRKSPLF「JOB余分／SELECT順序疑義」等）し、**その全てが
  主エージェントの原典照合で否定**された。委譲レビューが品質を上げるどころか誤誘導しかけた。
  → **原典・外部ソース照合を伴う検証は、サブ委譲では成立しない（委譲先の権限が落ちると幻覚指摘になる）**。
- **主エージェント自身の機械diffも偽陽性**: WRKSPLF の `JOB` を「原典に無い余分」と一度誤検出。原因は
  原典の複数行テーブル行（`JOB ジョブ名 単一値:*ALL` が次行に `オプショナル` を持つ）を grep が
  取りこぼしたこと。**原典の生テキストを直接 Read して確定**したことで解消。
  → grep要約への過信は禁物。一次ソースを直読する。

  パラメータ集合・required・定義済み値は一次ソースとの機械diffで確定する。
- （「上流で PJ内部仕様を確認」は #13 で起票済。重複回避。）

### ハーネス自体（→ aidev-* への提案・適用は人間）

- **委譲の但し書きを protocol §2.6 に追加**: 「**外部ソース／一次資料との照合を伴う検証（レビュー含む）は
  サブエージェントに委譲しない**。委譲先のツール権限が落ちると照合不能となり、幻覚的な指摘を生む。
  この種の検証は主エージェントが実施する」。今回の最大の学び。
- **`cl-command-def` / 関連レビューに同趣旨を明記**: 定義の正誤検証は IBM原典の生テキストを主エージェントが
  直読して行う。
- **aidev-batch の test硬ゲートの強化検討**: データ定義（JSON）では現状「JSON parse＋スキーマ整合」止まり。
  **「原典パラメータ表との機械diff（過不足・required）」を硬ゲートに含める**運用ガイドを足すと、
  生成物の事実誤りを着地前に機械的に弾ける（今回は手動で実施した分を定型化）。
