# 要件: lint core パッケージ（桁位置検査）

## 背景 / 課題

AS/400 の固定長ソース（RPG / DDS）は**桁位置に意味がある**。欄をまたいで書く・長さ欄を
左詰めにするといった誤りは、編集中は見た目で気付けず、**実機にコンパイルを投げて初めて**
`CPF7311` などで判明する。往復のコストが高く、自律ループ（編集→転送→コンパイル→修正）の
1 周あたりの時間を支配する。

一方で本 PJ は、この検査に必要な材料を**既に持っている**。プロンプター定義
（`resources/prompter/{dds,rpg}/…`）は欄ごとに `sourceStart` / `sourceLength` /
`required` / `attributes`（`characterSet` / `maxLength` / `numericOnly`）/ `options` を持ち、
いずれも IBM 原典と機械照合済みである。設計書
`docs/workflow/ibmi-dev-workflow.md`「4.3 役割 3」はこれを**ローカル層（オフライン・即時）**
として位置づけ、「世に存在せず、材料は本 PJ にのみある」(F7) と評価している。

現状この材料は F4 プロンプターと表示系からしか使われておらず、**検査には使われていない**。

## 目的 / ゴール

固定長ソースの桁位置の誤りを、**実機に接続せずその場で**指摘できる状態にする。
検査ロジックを VS Code に依存しない単一のモジュールに置き、次の 2 つの利用経路から使う。

1. **CLI** — CI で走らせられる lint コマンド（SARIF 出力）
2. **VS Code** — 編集中の入力時診断（既存の `diagnostics.ts` 経由）

同じロジックが両経路で動くことで、「エディタでは出ないが CI で落ちる」食い違いを構造的に防ぐ。

## スコープ

### 対象

- **lint core**: VS Code API に依存しない桁位置検査モジュール。
  プロンプター定義（既存）を規則の供給源とする。
- **対象言語**: RPG（ILE / RPG III）および DDS 全種別（PF / LF / DSPF / PRTF、`.mnudds` を含む）。
- **検査項目（4 種）**:
  1. 桁のはみ出し・行長超過
  2. 文字種違反（`numericOnly` / `characterSet` / 右寄せ必須）
  3. 必須欄の未入力
  4. 定義済み値以外の値
- **CLI**: ファイル/グロブを受けて検査し、**SARIF** を出力する。CI から単独で実行できる。
- **VS Code 連携**: 既存 `src/language/diagnostics.ts` に接続し、**入力時**
  （`onDidChangeTextDocument`）に診断を出す。

### 対象外

- **キーワード使用レベル検査**（DDS の file/record/field レベル）
  — backlog `workflow.md` P2 の別項目として切ってあるため、本作業では扱わない。
- **CL / `.cmd` の検査** — 自由形式で桁の規定が原典に無い（AGENTS.md「.cmd」節）。
  桁位置検査の対象にならない。なお `.clp` の既存 CL 診断（`parseClDocument`）は
  現状のまま残し、本作業では触らない。
- **実機層の検査**（`OPTION(*NOGEN)` / QTEMP コンパイル / `QCAPCMD`）
  — 設計書 4.3 の別レイヤ。接続を要するため本作業のオフライン層とは分ける。
- **自動修正（fix）** — 検出のみ。
- **CI ワークフロー（GitHub Actions）の整備そのもの** — backlog P3「CI を整える」の担当。
  本作業は「CI から呼べる形にする」までを負う。

## 機能要件

- 検査ロジックは **VS Code API・ネットワーク・実機接続のいずれにも依存しない**。
  ソース文字列と規則（プロンプター定義）だけで結果が決まる純粋な関数として成立する。
- 行ごとに**仕様書種別／DDS 種別を判定**し、その種別の桁定義を規則として適用する
  （RPG は 6 桁目のスペック文字＋方言、DDS はファイル種別＝拡張子）。
- 検出結果は、**ファイル・行・桁範囲・規則 ID・メッセージ・重大度**を持つ。
  桁範囲を持つことが本 lint の要（「どの欄か」が指摘の本体であるため）。
- CLI は **SARIF** を標準出力またはファイルに出す。検出の有無を**終了コード**で示す。
- VS Code 側は入力のたびに再評価し、`isInScopeDocument` の対象拡張子でのみ動く。
- 検査項目は**個別に無効化できる**（規則 ID 単位）。偽陽性が判明した規則を、
  パッケージ全体を止めずに切れるようにする。

## 非機能要件 / 制約

- **偽陽性ゼロを優先する**。正しいコードを警告するくらいなら検出しない。
  CI を落とす経路に乗るため、誤検出の害が見逃しの害を上回る。
  判定に確信が持てない規則は既定で無効にするか、実装しない。
- **規則は原典由来の既存定義から導く**。lint 用に桁や値を書き起こさない
  （AGENTS.md「同じ概念集合を複数箇所で列挙しない」）。
  ルーラー・プロンプター・lint で桁が食い違わないことを検査で担保する。
- **ランタイム依存を増やさない**。`vscode-extension` は現在 `dependencies: {}` であり、
  この状態を保つ。
- **入力時に動く速度**であること（1 ファイルの検査が編集を妨げない）。
- 既存のテスト経路（`npm test` / mocha tdd / `vscode-stub.js`）に乗せる。

## 完了条件 (受け入れ基準)

- [ ] 検査モジュールが `vscode` を import せずに動作する（機械的に検査できる形で担保）
- [ ] RPG（ILE / RPG III）と DDS 全種別のソース行に対し、上記 4 種の検査が動く
- [ ] CLI が SARIF を出力し、SARIF スキーマとして妥当である
- [ ] CLI が VS Code 非在の環境（`node` のみ）で動作し、検出有無を終了コードで返す
- [ ] VS Code で対象拡張子のファイルを**編集中に**診断が更新される
- [ ] **正常なソースに対する検出がゼロ**であることを、既存の検証済み資産
      （プロンプター往復検証の入力・原典の使用例など）を流して確認できる
- [ ] 規則 ID 単位で個別に無効化できる
- [ ] lint の桁定義がプロンプター／ルーラーの桁定義と一致することが検査で担保される
- [ ] 既存機能（F4 プロンプター・ルーラー・SOSI・補完・CL 診断）に退行がない

## 未確定事項 / 確認したいこと

いずれも **research 工程で解消すべき**もの（ユーザーへの質問では埋まらない）。

- **必須欄の材料が足りていない可能性**。既存定義 36 ファイル・346 欄のうち
  `required` が付いているのは **22 欄**のみで、**DDS-PF は 0 欄**。
  「必須欄の未入力」検査が成立する範囲を先に確定する必要がある。
- **`options` が制限なのか候補なのかが未確定**。`options` を持つのは 346 欄中 **26 欄**。
  AGENTS.md には「**列挙した値＝制限とは限らない**」（CL の `Rstd=NO` で 86 欄が該当し
  157 件を誤って弾いた）という教訓がある。DDS/RPG の定位置欄でも同じかを、
  原典の記述で欄ごとに判定してからでないと「定義済み値以外の値」は偽陽性を生む。
- **`specClassifier.ts` / `dialect.ts` が `vscode.workspace.getConfiguration` に依存**している
  （設定読みのみ）。VS Code 非依存に剥がす際の影響範囲（既存の利用元は
  `ruler.ts` / `positionResolver.ts` / `rpgCompletion.ts`）の確認が要る。
- **DDS の条件付け行・注記行・継続行の扱い**が未確定。注記行（7 桁目 `*`）や
  キーワードだけの行は定位置欄が空でも正しく、検査から除外する条件を定める必要がある。
- **I/O 仕様書の変種判定は行単体で決まらない**（F 仕様書 22 桁目の `E` に依存）。
  ファイル全体を見る必要があり、行単位 API では足りない可能性がある。
- **偽陽性ゼロを検証する材料**の特定。既存の `verify-prompter-roundtrip.mjs`(538 定義) や
  原典の使用例を流用できるかを確認する。
