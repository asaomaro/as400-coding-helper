# 要件: DDS 帳票（PRTF）の視覚編集 — DBCS 対応

## 背景 / 課題

DDS で書いた帳票（PRTF）は、**印刷して初めてレイアウトが分かる**。項目の桁位置は
39-44 桁に数字で書かれているだけで、紙面上でどこに出るかはソースを読んでも見えない。
SEU 時代には RLU（Report Layout Utility）があったが、VS Code には相当するものがない。

### 競合は存在するが、日本語で破綻する

IBM i Development Extension Pack には **IBM i Renderer**
（`halcyontechltd.vscode-displayfile` v0.1.3）が同梱されており、DSPF と PRTF の
プレビューを持つ。**配布されている VSIX を展開して実装を直読**したところ:

| 項目 | 実際 |
|---|---|
| DBCS / SOSI の処理 | **0 件**（`0x0E` / `0x0F` / `ccsid` / `dbcs` / `charCodeAt` すべてヒットなし） |
| 桁の計算 | `line.substring()` と `value.length` ＝ **JS の文字数** |
| 対応種別 | DSPF・PRTF 両方（`onLanguage:dds.dspf` / `dds.prtf`、132 桁の紙面） |
| 書き戻し | **無し。閲覧専用**（`writeFile` / `applyEdit` が 1 件も無い） |

つまり `顧客名` は JS では 3 文字だが実機では 6 桁を占めるため、**日本語を含む帳票は
プレビューの見た目と実際の印刷結果がずれる**。フィールドの重なりも検出できない。

### 上流に投げても解決しない見込み

同種の「プレビューの桁が実態と合わない」指摘である `vscode-ibmi#2357`
（DSPF プレビューが 80 桁固定で DS4 が切れる）は、メンテナが反応し利用者が
サンプルを提供し 3 人目が追随した上で、**1 年 8 か月 OPEN のまま**。
`vscode-ibmi#717`（多言語対応）も協力者が名乗り出たまま `ja` は入っていない。
DBCS は日本語圏だけの問題なので、さらに後回しになる可能性が高い。

加えて `vscode-displayfile` は**リポジトリ (0.0.1) と配布版 (0.1.3) が同期しておらず**
（README に明記）、PR がマージされても利用者に届く経路が読めない。

### 本 PJ には材料が揃っている

- `dbcsShiftMarkers.ts` の `isDbcsCodePoint`（全角の判定）と SO/SI の仮想表示
- DDS-PRTF の桁定義（原典から生成・ルーラーと共有・機械検査済み）
- DDS キーワードの一覧と使用レベル判定
- 固定長ソースへの**書き戻しの実績**（F4 プロンプターの `applyChanges`）

## 目的 / ゴール

**帳票のレイアウトを VS Code 上で見ながら組めるようにする。**
日本語を含む帳票でも、画面に見えるものが実機の印刷結果と一致する。

閲覧だけでなく**画面上で項目を動かして DDS に反映できる**（RLU 相当）ところまでを狙う。
競合が閲覧専用であること、および編集こそが本来の需要であることによる。

## スコープ

### 対象

- **PRTF（印刷装置ファイル）** の視覚編集。
- **紙面のプレビュー**: A 仕様書を読み、項目・定数を紙面上の桁行位置に描く。
- **DBCS を実機と同じ桁で扱う**（全角＝2 桁、SO/SI の分も数える）。
- **書き戻し**: 画面上で項目の位置を変えると、ソースの位置欄（39-44 桁）に反映する。
- ソースの行と描画の項目を相互に対応づける（どの行がどれか分かる）。

### 対象外

- **DSPF（表示装置ファイル）** — 別作業として後続で扱う。A 仕様書の解析は共通に
  できる見込みだが、条件付け・標識・サブファイルなど画面固有の要素が多く、
  同時に抱えると初版が終わらない。
- **物理/論理ファイル（PF/LF）** — 視覚化の対象ではない。
- **実機との通信** — ローカルのソースだけで完結する（本 PJ の既定方針）。
- **印刷プレビューの完全再現**（フォント・実寸・PDF 出力）。桁と行の位置関係が
  分かることが目的で、印刷そのものの再現は狙わない。
- **項目の新規作成・削除** — 初版は既存項目の移動に絞る。
- IBM i Renderer の置き換え。**共存**を前提とする。

## 機能要件

- `.prtf` を開いた状態から、**コマンドで**帳票プレビューを開ける。
- プレビューは**紙面の桁数・行数**を DDS の指定（`PAGESIZE` 等）から決める。
  指定が無ければ既定値を使う。
- 項目・定数を、ソースの位置指定（39-44 桁）どおりの桁行に描く。
- **DBCS を含む値の幅を実機の桁で計算する**。全角 1 文字＝2 桁、
  DBCS の連なりの前後に SO/SI が 1 桁ずつ。
- 項目が**重なっている**ことが分かる（実機では作成に失敗するか印刷が崩れる）。
- 画面上で項目を動かすと、**ソースの位置欄に書き戻す**。
  書き戻しは位置欄だけを変更し、**行の他の桁を壊さない**。
- ソースの行と描画の項目が相互にたどれる。
- **既存の IBM i Renderer と同時に導入しても衝突しない**。
  本 PJ は言語登録をせず拡張子で判定する既定方針に従い、コマンドも別にする。

## 非機能要件 / 制約

- **桁の真実源は既存の DDS 桁定義**（`resources/navigation` / プロンプター定義）。
  ここで桁を書き起こさない（AGENTS.md）。ルーラー・プロンプター・lint と食い違わせない。
- **DBCS の幅の判定は既存の `isDbcsCodePoint` を使う**。SOSI 表示と同じ規約にする。
- **ランタイム依存を増やさない**（`vscode-extension` の `dependencies: {}` を維持）。
  競合は描画に `konva` を使っているが、同じ選択をするとは限らない。
- **書き戻しは既存の流儀に合わせる**。プロンプターの `applyChanges` が
  「指定した桁だけを置き換え、他を壊さない」形を確立しているので、それを踏襲する。
- 日本語 UI（本 PJ の既定）。

## 未確定事項 / 確認したいこと

いずれも **research 工程で解消すべき**もの。

- **ローカルのソースに SO/SI は存在しない**。`docs/src/DBCSSAMP.pf` を生バイトで
  確認したところ、日本語は UTF-8 のまま（`346 274 242` = `漢`）で `0x0E`/`0x0F` は
  無かった。`dbcsShiftMarkers` は SO/SI を**仮想的に**表示している。
  よって「実機の桁」を出すには、**ソースには無い SO/SI の分を計算で足す**必要がある。
  この規約が既存の SOSI 表示・ルーラー・lint とどう整合するかを確定する。
- **紙面の大きさをどう決めるか**が未確定。`PAGESIZE` の意味・既定値、`CPI`/`LPI` が
  桁に効くのかを原典で確認する必要がある。
- **桁送りキーワードの扱い**。`SPACEB`/`SPACEA`/`SKIPB`/`SKIPA` は行位置を相対的に
  動かすため、位置欄（39-44）だけでは行が決まらない可能性がある。原典の確認が要る。
- **位置欄が空の項目**がある（競合のコードに「印刷装置ファイルで Y 位置が無い」旨の
  コメントがある）。その場合どこに出るのかを原典で確定する。
- **書き戻しの安全性**。位置欄の桁だけを置き換えて他を壊さないことを、
  既存の `applyChanges` がどう担保しているかを読む。
- **描画の実現手段**が未確定。ランタイム依存ゼロの制約下で WebView に何を使うか
  （競合は `konva`）。素の HTML/CSS で足りるかを見極める。
- **IBM i Renderer との共存**が実際に成り立つかは未検証。あちらは `dds.prtf` の
  languageId で発火するため、`barrettotte.ibmi-languages` が入っている環境では
  両方が動く。干渉の有無を確かめる。
