# 決定記録

## D1: 桁送りキーワードのレベル別の意味を原典で確定した（T8・リスク潰し）

- 背景: `design.md` D3 のリスクとして「レコード・レベルの `SPACEA` が
  『レコードの最後の項目の後』に効くのか『各項目の後』に効くのか」が未確定だった。
  サンプル `CUSTRPT.prtf` の `R DETLINE SPACEA(1)` が該当し、誤ると明細行の送りが
  全部ずれる。レイアウト解決を書く前に確定させる必要があった。
- 決定: **レコード・レベルはレコードの全行の前後、フィールド・レベルはその項目の前後**。
  4 キーワードとも対称。

  | キーワード | レコード・レベル | フィールド・レベル |
  |---|---|---|
  | `SPACEA(n)` | レコードの**すべての行が印刷された後**に n 行送る | **そのフィールドが印刷された後**に n 行送る |
  | `SPACEB(n)` | レコードの行の**手前**で n 行送る | そのフィールドを含む行の手前で n 行送る |
  | `SKIPA(n)` | レコードの**すべての行の後**に行 n へ飛ぶ | そのフィールドの後に行 n へ飛ぶ |
  | `SKIPB(n)` | レコード印刷に関連する**任意の行の手前**で行 n へ飛ぶ | そのフィールドが印刷される前に行 n へ飛ぶ |

  原典（`印刷装置ファイルの SPACEA (印刷後スペース) キーワード`）:
  > このキーワードを**レコード・レベル**で指定した場合には、該当のレコードの
  > **すべての行が印刷された後**で行送りが行われます。このキーワードは、
  > レコード・レベルで 1 回、そして**各フィールドについて 1 回ずつ**指定することができます。
  > このキーワードを**フィールド・レベル**で指定した場合には、
  > **そのフィールドが印刷された後**で行送りが行われます。

- 理由 / 代替案: 推測で「各項目の後」に倒す案もあり得たが、原典が明示している以上
  照合して確定するのが本 PJ の規約（AGENTS.md）。実際に原典は
  「レコード・レベルで 1 回、そして各フィールドについて 1 回ずつ指定できる」と
  **両方が併存しうる**ことまで書いており、推測では到達できなかった。
- 影響: `prtfLayout` の印刷カーソルは**レコードの開始・終了**と**各項目の前後**の
  4 か所で更新する。`design.md` D3 の表を上記で確定させる。

## D2: 桁送りキーワードと行番号の併用は「原典が誤りと明示している」ので検出対象にする

- 背景: D1 を調べる過程で、原典が 4 キーワードすべてについて同じ制約を書いていることが
  分かった（設計時には見えていなかった）。
- 決定: 次の 2 つを**レイアウト解決の警告として検出する**。
  1. **行番号（39-41 桁）のあるレコードで `SPACE*` / `SKIP*` を使っている**
     > このキーワードは行番号の指定 (39 から 41 桁目) のあるレコードについては
     > **無効です。(行番号にはエラーを示すフラグが付けられます。)**
  2. **レコード様式に `BOX` / `ENDPAGE` / `GDF` / `LINE` / `OVERLAY` / `PAGSEG` /
     `POSITION` があるとき**に `SPACE*` / `SKIP*` を使っている
     > …も指定されている場合には、フィールド・レベルまたはレコード・レベルでは無効です。
- 理由 / 代替案: 黙って片方を採る（行番号を優先する等）と、**実機ではエラーになる
  ソースが画面上は正しく見える**。この作業の目的が「実機と一致する見た目」である以上、
  食い違いは見せなければならない。
- 影響: `PrtfLayout` に警告の一覧を持たせる。`spec.md` の型に
  `diagnostics: readonly LayoutDiagnostic[]` を足す（`overlaps` / `overflows` も
  ここに寄せるか、別に持つかは実装時に決める）。

## D3: 行番号もスキップ／スペースも無い場合の「2 重印刷」を警告にする

- 背景: 原典の注記:
  > **注**: 行番号を使用せず、スキップ・キーワードもスペース・キーワードも
  > 指定しなかった場合には、**2 重印刷 (重ね打ち) が行われることがあります。**
- 決定: この状態を検出して警告として示す。**エラーにはしない**
  （原典も「行われることがあります」であり、意図的な重ね打ちもあり得る）。
- 理由: 帳票が 1 行に潰れる典型的な誤りで、プレビューで見れば一目で分かるが、
  「なぜそうなるか」を示せた方が親切。
- 影響: D2 と同じ警告の仕組みに乗せる。

## D4: レイアウト解決を「行ごと」から「論理単位ごと」に変えた

- 背景: `design.md` は「行を頭から走査する」形で設計していたが、実装して
  `CUSTRPT.prtf` を流したところ**見出しと明細が同じ 3 行目に重なった**。
  原因は DDS の**キーワード継続行**を独立した行として扱っていたこと。

  ```
       A          R HEADING                   SKIPB(1)
       A                                      SPACEA(2)   ← HEADING のキーワード
       A                                    30'顧客一覧表'
  ```

  2 行目の `SPACEA(2)` は **`R HEADING` のレコード・レベルのキーワード**で、
  独立した桁送りではない。行ごとに処理すると桁送りの持ち主を取り違える。
- 決定: 行を**論理単位**（レコード様式＋その継続行 / 項目＋その継続行）に
  まとめてから走査する。キーワードだけの行は直前の単位に連結する。
- 理由 / 代替案: 「前の行の種類を覚えておいて分岐する」形でも書けるが、
  キーワードが複数行に分かれる場合（DDS では珍しくない）に、キーワードの
  連結と桁送りの適用が別々の場所に散る。**単位にまとめてから 1 回だけ
  解釈する**方が、キーワードが何行に分かれていても同じ経路を通る。
- 影響: `design.md` の走査の図（行ごとの分岐）は**論理単位ごと**に読み替える。
  `toLogicalUnits` を通してから `resolvePrtfLayout` の本体が回る 2 段になった。
  正しい結果:
  ```
    行 1 桁 30  '顧客一覧表'  12 桁          [HEADING]
    行 3 桁  5  CUSTNO       幅不明(reference) [DETLINE]
    行 3 桁 15  CUSTNM       幅不明(reference) [DETLINE]
    行 3 桁 50  CUSTAM       幅不明(reference) [DETLINE]
  ```
  この誤りは**サンプルを流さなければ気づけなかった**（型検査も単体テストも
  通っていた）。回帰テストで固める。
