AI 開発ワークフローの **retro（振り返り）工程**（任意）。ワークフローの締め。
完了した作業を振り返り、学びと改善提案を構造化して残す。**提案までを担い、適用は人間が行う**
（特にハーネス自体の改変は自動で行わない）。

**開始前に共通プロトコル `../aidev-00-start/protocol.md` を読み、その規約に従うこと。**

## 位置づけ

- 任意工程（番号末尾以外の独立番号 90、deliver の後の締め）。毎回は不要。
- 主軸は「作業の振り返り」。その出力の一カテゴリとして「ハーネス自体の改善提案」を含む。
- 起動は基本ユーザー指定（AI検知による自動推奨は行わない）。

## 前提

- 作業が一通り完了していること（基本は deliver 済み）。

## 入力

- 対象フォルダの全成果物（`requirement.md` 〜 `decisions.md`、`state.yml`）。
- `metrics.yml`（実施日時・各工程の経過時間・手戻り回数・リードタイム・件数指標の導出元）。
- `review.md`（レビュー指摘の内容。再発パターンの分析に使う）。
- 差し戻しの履歴・`decisions.md`（手戻りの痕跡）。
- 変更差分・テスト/レビュー結果。

## 出力

対象フォルダに `retro.md` を生成する。改善提案は次の3カテゴリに仕分ける。

## 手順

1. protocol.md「1. 対象作業の特定」に従い対象フォルダを確定する。
2. 作業全体を振り返る。観点:
   - うまくいった点 / うまくいかなかった点
   - 手戻り（差し戻し）の原因と、前段で防げたか
   - 見積もりと実績の差、想定外だったこと
3. **レビュー指摘・メトリクスの分析**（この作業の範囲で）。次を改善提案の根拠にする。
   - `review.md`：指摘の傾向を読む。同種の指摘の反復、本来 spec/test で防げた指摘、観点の抜け。
   - `metrics.yml`：手戻り回数・差し戻し回数・件数指標から、弱い工程を特定する。
   - 各観察を「どの段で防げたか」で分類する（上流工程 / 規約 / ハーネス）。
   - ※対象はこの works のみ（複数作業を横断した再発分析は retro の範囲外。別途行う）。
4. 学びと 3 の観察を**改善提案**に変換し、3カテゴリに仕分ける。
   - **製品 / コード**: 残課題・技術的負債 → 新 issue 化の候補
   - **PJ プロセス / 規約**: コーディング規約・レビュー観点の追補 → AGENTS.md への反映案
     （review.md の反復指摘・観点抜けはここに繋げる）
   - **ハーネス自体**: 工程・ゲート・protocol の不備 → `aidev-*` への変更提案（提案のみ）
5. 下記テンプレートに沿って `retro.md` を記述する。
6. protocol.md「3. 工程終了プロトコル」に従って終了する。
   - 締めの工程のため、遷移確認は `承認して完了` とする。
   - 採用された提案は、ユーザーの指示で別途（新 issue / AGENTS.md 編集 / 基盤改修）対応する。

## retro.md テンプレート

```markdown
# 振り返り: <タイトル>

## サマリ
<この作業の結果と総括>

## うまくいった点
- <継続したいこと>

## 課題 / 手戻り
- <発生した問題と原因。前段で防げたか>

## 改善提案
### 製品 / コード（→ issue 候補）
- <残課題・負債>

### PJ プロセス / 規約（→ AGENTS.md）
- <規約・観点の追補案>

### ハーネス自体（→ aidev-* への提案・適用は人間）
- <工程・ゲート・protocol の改善案と理由>
```

## 完了の目安

- 手戻りの原因が言語化され、再発防止の提案に変換されている。
- 改善提案が3カテゴリに仕分けられ、次アクションが明確（提案止まりでよい）。
