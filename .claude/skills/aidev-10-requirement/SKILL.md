AI 開発ワークフローの **requirement（要件定義）工程**を実行する。
開発の起点。ユーザーの要望を整理し「何を・なぜ作るか」を `requirement.md` として確定する。
実装手段（どう作るか）には踏み込まない（それは spec 以降）。

**開始前に共通プロトコル `../aidev-00-start/protocol.md` を読み、その規約に従うこと。**

## 前提

- 新規作業として `aidev-00-start` から起こされていること（`.aidev/current` が設定済み）。
  未設定なら工程を実行せず `aidev-00-start` を案内する（protocol.md「1. 対象作業の特定」）。
- 前提となる成果物は無い（この工程が起点）。

## 入力

- ユーザーの要望・課題・背景。
- 必要に応じて既存コードや PJ 固有ルールを参照してよい。

## 出力

対象フォルダ（`.aidev/works/<YYYYMMDD-slug>/`）に `requirement.md` を生成する。

## 手順

1. protocol.md「1. 対象作業の特定」に従い、`.aidev/current` から対象フォルダを確定する。
2. ユーザーの要望をヒアリングし、曖昧な点は質問して埋める。
   - 解決したい課題・背景（なぜ）
   - 対象ユーザー／利用シーン
   - 完了条件（何が出来たら完成か）
   - スコープ外（やらないこと）の明示
   - 制約（性能・互換性・PJ のドメイン固有の前提があれば）
3. 下記テンプレートに沿って `requirement.md` を記述する。実装方法は書かない。
4. **調査不足の自己評価（research 推奨判定）**：protocol.md「4.5」に従い、次のいずれかに該当するか確認する。
   - 「未確定事項」に、ユーザーへの質問ではなく**調査で解消すべき**項目が残っている
   - requirement が**未検証の既存挙動・既存コード**に依存している
   - 技術的実現性の**仮定**が含まれる
   - 変更が**横断的**になりそう（影響範囲が広い兆候）
   該当すれば、次の遷移ゲートの選択肢に `承認して research(任意) を挟む`（推奨）を加え、推奨理由を添える。
5. protocol.md「3. 工程終了プロトコル」に従って終了する
   （提示 → 承認 → `state.yml` 更新 → 次工程 `spec`、または推奨時は `research` へ進むか確認）。

## requirement.md テンプレート

```markdown
# 要件: <タイトル>

## 背景 / 課題
<なぜこれが必要か>

## 目的 / ゴール
<達成したい状態>

## スコープ
### 対象
- <やること>

### 対象外
- <やらないこと>

## 機能要件
- <満たすべき機能を箇条書き>

## 非機能要件 / 制約
- <性能・互換性・PJ のドメイン固有の制約>

## 完了条件 (受け入れ基準)
- [ ] <検証可能な条件>

## 未確定事項 / 確認したいこと
- <あれば。spec で解消する>
```

## 完了の目安

- 「なぜ・何を」が第三者に伝わり、完了条件が検証可能な形で書けている。
- 実装の詳細に踏み込んでいない（手段の決定は spec へ送る）。
