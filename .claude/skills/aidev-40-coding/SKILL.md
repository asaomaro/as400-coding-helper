AI 開発ワークフローの **coding（実装）工程**を実行する。
plan / tasks に沿ってコードを書く。`tasks.md` の未チェック項目を 1 つずつ実装し、
完了したらチェックを付けて進捗を更新する。

**開始前に共通プロトコル `../aidev-00-start/protocol.md` と、PJ 固有のコーディング規約を読むこと。**

## 前提

- `plan.md` と `tasks.md` が存在すること。無ければ実行を中止し、plan 工程を促す。

## 入力

- 対象フォルダの `tasks.md` / `plan.md` / `spec.md`。
- 既存コードと PJ 固有のコーディング規約（AGENTS.md 等。存在すれば優先して従う）。

## 出力

- 実装コード（リポジトリ本体への変更）。
- 進捗を反映した `tasks.md`（完了タスクにチェック）。
- 設計から外れた判断をした場合は `decisions.md` に記録。

## 手順

1. protocol.md「1. 対象作業の特定」に従い対象フォルダを確定。「2. 前提チェック」に従い `plan.md` / `tasks.md` を確認。
2. `tasks.md` の未チェック項目から、依存関係を満たす次のタスクを選ぶ。
3. PJ のコーディング規約に従って実装する。周辺コードのスタイルに合わせる。
4. タスク完了ごとに `tasks.md` の該当項目にチェックを付ける。
5. 仕様から逸脱する判断をした場合は `decisions.md` に理由を残す（後続工程・レビューのため）。
6. 区切り（全タスク完了、または一区切り）で、protocol.md「3. 工程終了プロトコル」に従って終了する（次工程: `test`）。
   承認は `aidev approve coding tasks_done=<チェック済みタスク数>`（protocol.md「3.」「8.」）。

## 留意点

- 1 度に多くのタスクをまとめて進めず、検証可能な単位で刻む。
- 全タスクが終わっていなくても、区切りで中断してよい（`tasks.md` のチェック状態がレジューム点になる）。
- review / test からの差し戻しでこの工程に戻ることがある。その場合は指摘に対応するタスクを処理する。

## decisions.md テンプレート

設計から逸脱した判断や、後続工程・レビューに影響する決定を**追記式**で残す。
1 決定 = 1 エントリ。ID は `D1`, `D2`… と連番。既存エントリは編集せず、新しい決定を追記する。

```markdown
# 決定記録

## D<連番>: <決定の要約（一文）>

- 背景: <なぜこの判断が必要になったか>
- 決定: <何を決めたか>
- 理由 / 代替案: <採用理由。退けた代替案があれば添える>
- 影響: <spec/plan など他成果物への影響。差し戻し起点なら経緯も>
```

## 完了の目安

- `tasks.md` の対象タスクが全てチェック済み。
- spec の受け入れ基準を満たすコードが書けている（検証は test 工程）。
