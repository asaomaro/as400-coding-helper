AI 開発ワークフローの **research（事前調査）工程**（任意）。
spec を書く前に、既存コード・外部情報・実現性を調査し、不確実性を事実で解消する。
手戻り（誤った spec → 差し戻し）の予防が目的。

**開始前に共通プロトコル `../aidev-00-start/protocol.md` を読み、その規約に従うこと。**

## 位置づけ

- 任意工程（番号末尾5、requirement と spec の間に差し込む）。毎回は不要。
- 起動経路は2つ（protocol.md「4.5」）: ユーザー指定、または requirement 終了時の AI検知＋推奨。
- 重く対話の少ない調査が多いため、サブエージェントへの委譲（protocol.md「2.6」）と好相性。

## 前提

- `requirement.md` が存在すること。

## 入力

- 対象フォルダの `requirement.md`（特に「未確定事項」）。
- 既存コード、外部ドキュメント、PJ ルール。

## 出力

対象フォルダに `research.md` を生成する（spec の入力になる）。

## 手順

1. protocol.md「1. 対象作業の特定」に従い対象フォルダを確定。「2. 前提チェック」で `requirement.md` を確認。
2. 調査すべき問い（unknowns）を列挙する。典型例:
   - 既存コードの実装把握 / 影響範囲（どのコードが対象挙動を担うか、変更が波及する箇所）
   - **出力先の仕様（生成・変更する対象のスキーマ・型・フォーマット・既存実装）**。
     成果物が既存の型/スキーマに準拠する必要がある場合、その定義を必ず読む
     （外部情報源だけでなく、PJ 内の出力先仕様を確認しないと後段で手戻りになりやすい）。
   - 技術的実現性（API・ライブラリで実現可能か）
   - requirement の未確定事項のうち、調査で解消すべきもの
3. 各問いを調査し、**事実**で答える（推測と区別する）。重い調査はサブエージェントに委譲してよい。
4. 下記テンプレートに沿って `research.md` を記述する。設計判断（how）は spec に回す。
   既存構造・呼び出し関係・影響範囲は、明確になるなら mermaid で図示する（protocol.md「9.」）。
5. protocol.md「3. 工程終了プロトコル」に従って終了する（次工程: `spec`）。

## research.md テンプレート

```markdown
# 調査: <タイトル>

## 調査の問い
- Q1: <明らかにしたいこと>

## 判明した事実
- F1: <Q1 に対する事実。根拠（ファイル/行・出典）を添える>

## 影響範囲
- <変更が波及する箇所・関連コンポーネント>

## 実現性 / リスク
- <技術的に可能か。制約・懸念>

## spec への申し送り
- <設計時に踏まえるべき点。残った未確定事項>
```

## 完了の目安

- requirement の「調査で解消すべき未確定事項」が事実で埋まっている。
- spec が誤った前提で書かれるリスクが下がっている。
