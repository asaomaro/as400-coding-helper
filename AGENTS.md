# Overview

- VSCode上でAS400のコーディング(RPG,RPGLE,CLP,DDS,PRTF,CMD)をサポートするVSCode拡張機能を開発するPJ

## 言語

- TypeScript

## サポートする機能

### プロンプトによるコード入力補助機能

- SEUによる編集操作でF4キーから呼び出されるプロンプトウィンドウによる入力機能をサポート
- RPGLE(ILE RPG)は、固定長フォーマットのみをサポートし、自由記述(free format)には対応しない
- プロンプトウィンドウの種類ごとに、JSONフォーマットの外部ファイルで設定を定義できる設計
  - RPG,RPGLEソースの仕様書(D-Spec,C-Spec etc)単位
    - C-Specなど、古い命令と新しい命令(ex. MOVEL & EVAL)で、固定長フォーマットの記述位置が異なるケースでは、別々の定義ファイルを使用する
  - CLコマンドは、コマンドごとに定義ファイルを用意する
- 各パラメータの属性(文字、数値)や必須入力(required)を定義でき、validationが可能
- コマンドにはヘルプを設定でき、プロンプトのヘルプボタンからコマンドについてのヘルプが参照できる
  - 各パラメータにも個別にヘルプを設定でき、フォーカス中にF1キーをクリックすることでパラメータのヘルプを参照できる
- コマンドの可変パラメータに対応し、プロンプトウィンドウ上の操作で、入力項目の増減が可能
- グルーピングされたコマンド(ex. FILEパラメータは、FILE(LIBL/FILE) のようにライブラリーとファイルの2つのパラメータを持っている)を扱えるよう、定義でグルーピングを設定できるようにする
  - プロンプトウィンドウ上でもグルーピングされている事が分かるデザインで表現する
- ソースファイル上からF4キーをクリックすることで、VSCodeの別タブでヘルプウィンドウが表示できる
  - ソースファイルでフォーカスの当たっていた行のコードの内容がプロンプトウィンドウに表示される

### ルーラー表示

- SEUによる表示では、固定長フォーマットの桁位置が分かるようにルーラーが表示されている
- VSCodeによる編集時にもフォーカスしている行の上側にルーラーを表示されるようにする
- ルーラーはコードを書き換えずに入力補助として表示させる
- ルーラー表示は、特定のファイル拡張子(rpg,rpgle,sqlrpgle,sqlrpg,clp,clle,pf,lf,dspf,prtf,mnudds,dds,cmd)の場合に表示させる
- ルーラー表示のオンオフを、VSCodeのステータスバークリックで切り替えられるようにする

### 制御コード(SOSI)表示

- SEUにる表示では、DBCSの前後にSO(shift out)とSI(shift in)が挿入される
- VSCodeでの表示では制御コードが表示されず、正しい桁位置で表示がされない
- VSCode上でも正しい桁位置で表示されるよう、DBCS前後にSOを'{'、SIを'}'で表示させる
- '{','}'はコードを書き換えずに入力補助として表示させる
- '{','}'表示は、特定のファイル拡張子(rpg,rpgle,sqlrpgle,sqlrpg,clp,clle,pf,lf,dspf,prtf,mnudds,dds,cmd)の場合に表示させる
- 制御コード表示のオンオフを、VSCodeのステータスバークリックで切り替えられるようにする

## 開発時の検証規約

- **原典から機械的に決まる成果物は、LLM に書かせずスクリプトで生成する**。CL プロンプター定義は
  `docs/origin/generate-cl-definitions.mjs` が生成し、`docs/origin/verify-cl-definitions.mjs` が
  原典と突き合わせる（差分ゼロが受け入れ条件）。**JSON を手で直さず生成スクリプトを直す**こと。
  理由: 原典の情報にはマークアップにしか存在しないもの（省略時値の下線、修飾子/要素の別）があり、
  テキスト化して LLM に読ませると必ず落ちる。実際、過去の一括生成では 553 件の `defaultValue` が
  欠落し、13 件の `required` が食い違い、35 コマンドで入力欄の名前が重複していた。
- プロンプター定義(CL/RPG)など**原典のある成果物の正誤は、原典の生テキストを直接照合して確定する**。
  要約・知識・grep の取りこぼしに頼らない。パラメータ集合・必須/型・定義済み値は、IBM原典(または
  固定長フォーマットリファレンス)の生テキストと**機械的に突き合わせ**て過不足・誤りを弾く。
- この照合は**サブエージェントに委譲せず主エージェントが実施する**（委譲先のツール権限が落ちると
  原典照合ができず、知識ベースの幻覚的な指摘を生むため）。

- **ボット遮断の検出は本文の先頭だけを見る**。コマンドのページはエラー・メッセージ
  一覧を持ち、そこに `CPF2513 Message queue &1 cannot be displayed.` のような文が入る。
  本文全体を走査すると正しいページを遮断と誤判定する（`DSPMSG` の英語版で実際に踏み、
  取得できていないのに「本文が短い」と誤った理由が出ていた）。
- **原典の使用例が誤っていることがある**。`DLTQMFORM` の例は日英とも
  `DLTQMFORM QMQRY(...)` だが、実機は `CPD0043『Keyword QMQRY not valid』`を返す
  （正しくは `QMFORM`）。定義を原典の誤りに合わせず、**実機で確かめてから例の方を
  対象外にする**（`verify-cl-roundtrip.mjs` の `BROKEN_EXAMPLES`。根拠を必ず添える）。

### .cmd（コマンド定義ソース）

- `.cmd` に書くのは **CL コマンドではなくコマンド定義ステートメント**
  （`CMD` / `PARM` / `ELEM` / `QUAL` / `DEP` / `PMTCTL`）。`CRTCMD` を定義に足しても
  `.cmd` の入力補助にはならない。
- 原典は CL コマンドと同じ場所（`cl/<名前>.htm`）にあり、書式も構文も同じなので
  取得と生成は同じ経路を通す。ただし**出力先は `prompter/cmd/{lang}/` に分ける**。
  混ぜると CL のプロンプターに `PARM` や `QUAL` が出る。
- 言語 ID は `cmd`。言語登録はせず拡張子で判定する（表示系と同じ方針）。
  解析・書き戻しは CL と同じ経路（`positionResolver` / `applyChanges` の CL 分岐）。
- **桁の規定は原典に無い**。原典の使用例も桁が揃っておらず（文の開始が 1/7/8 桁と
  ページごとにばらばら）、実機も 1 桁目から書けばコンパイルが通る。つまり自由形式。
  ルーラーの `ラベル 1-13 / 文 14 / パラメータ 25` は **SEU の書き方であり、
  プロンプターが書き出す形**。原典由来ではないので、原典照合の対象にはしない。
- **73-80 桁は無視されない**。RPG/COBOL の「73 桁目以降は無視」は CL / .cmd には
  当てはまらない。実機で確認済み（`.cmd` は 73 桁目に置いた `LEN(1)` が効き、
  CL は 73 桁目に置いた不正キーワードが `CPD0043` になる）。
  プロンプターの 72 桁折り返しは安全側に倒しているだけで、桁上限ではない。

### 値そのものがコマンドのパラメータ（プロンプター内プロンプター）

- `SBMJOB CMD(...)` や `IF THEN(...)` のように、**値が CL コマンドそのもの**の
  パラメータがある。SEU ではその欄でさらに F4 が押せる（F4 in F4）。
- 判定は **原典のパラメータ表の「選択項目」欄が `コマンド・ストリング`
  (英語版 `Command string`) であること**。本文にも同じ語が出るので本文で
  探さない。該当は 6 件（`SBMJOB CMD` / `IF THEN` / `WHEN THEN` / `ELSE CMD` /
  `OTHERWISE CMD` / `MONMSG EXEC`）で日英一致。定義では `valueKind: "command"`。
- 欄に入るのは**値であってソース行ではない**。ラベル欄の桁揃えや 72 桁での
  折り返しを持ち込むと余計な空白が入るため、`buildClCommandBody`（素の 1 行）を
  使う。ソース行用の `buildClCommandText` と取り違えないこと。

### languageId / アクティベーション変更時の下流波及チェック

- VSCode 拡張では `languageId` が**診断・キーバインド・スニペット・補完などの発火条件**を兼ねる。
  そのため `contributes.languages[].extensions`（拡張子↔言語の関連付け）・`activationEvents`・
  言語登録そのものを変更する際は、**その languageId に紐づく既存 contributes に副作用が出ないか**を
  spec / review 段で必ず点検する。「拡張子を増やす」だけのつもりが言語機能ごと波及するのが典型の落とし穴。
- 具体的な確認観点（最低限）:
  - その拡張子に**意図しない診断が走らないか**（例: `.cmd` を `cl` に同居させると CL 診断 `parseClDocument` が発火）。
  - その拡張子に**意図しない編集キーバインドが付かないか**（例: `.dds/.dspf/.prtf` に RPG のコメントトグル/タブナビ）。
  - 補完・スニペット・フォーマッタ等、languageId 連動の他機能への波及。
- **表示系（ルーラー / SOSI）の有効化は原則 languageId 非依存（拡張子判定）で行う**。表示有効範囲の拡大を
  目的に言語登録を広げると言語機能まで巻き込むため、表示系は `fileScope.ts` の拡張子判定＋`onStartupFinished`
  起動で適用し、言語登録は「その言語の機能を本当に適用してよい拡張子」だけに限定する
  （`.rpg`→`rpg-fixed` は妥当、`.pf/.lf/.dspf/.prtf/.dds/.cmd` は言語登録しない、が現行の判断）

- **DDS のファイル拡張子は `.dds` ではない**。実務ではオブジェクトの種類ごとに
  `.pf`(物理ファイル) / `.lf`(論理ファイル) / `.dspf`(画面ファイル) / `.prtf`(印刷ファイル)
  を使う。`.dds` も対象に残してあるが、それは保険であって主たる形ではない。
  対象拡張子は `src/utils/fileScope.ts` の `TARGET_EXTENSIONS` が単一の真実。

- **SQL 組み込み・ILE 版の拡張子を落とさない**。`.sqlrpgle`(SQL組み込み ILE RPG)、
  `.sqlrpg`(SQL組み込み RPG III)、`.clle`(ILE CL) はいずれも固定長ソースで、
  `.rpgle` / `.clp` と同じ扱いが要る。拡張子判定は `fileScope.ts` だけでなく
  **`ruler.ts` の specFamily / `positionResolver.ts` の言語判定 / `dialect.ts` の
  方言マップ**にもあり、片方だけ足すとルーラーやプロンプターが効かない。
  実機(pub400)のソースタイプ分布でも RPGLE 3295 / CLP 1086 / CLLE 219 /
  SQLRPGLE 209 と、いずれも実使用がある。

- **対象に含めないと決めたもの**（実機のソースタイプ分布と実物を見て判断）:
  - `BND`(バインダー言語) … 固定長ではない。実物は C 風のコメントで始まる
    自由形式（`STRPGMEXP`/`EXPORT`/`ENDPGMEXP`）で、桁位置に意味が無く
    ルーラーが機能しない。PJ の対象言語にも含まれない。
  - `MNUCMD` … メニューのコマンド定義側。DDS ではなくコマンドの羅列。
  - `RPG36`/`RPG38`/`CBL36` … System/36・38 互換の移行資産。現代の開発では不要。
  - `CBLLE`/`CBL`/`C`/`CPP`/`PLI` … 件数は多いが本 PJ の対象言語ではない。
  なお `MNUDDS` は **DSPF と同じ A 仕様書の固定長**であることを実物で確認したため
  対象に含めている（6 桁目が `A`、キーワードは 44 桁目から）。。

### DDS の扱い

- DDS は同じ A 仕様書でも **用途（物理/論理・表示装置・印刷装置）で桁の意味もキーワードも
  変わる**。原典もファイル種別ごとに別ページ（`rzakb` / `rzakc` / `rzakd`）に分かれており、
  定義も種別ごとに持つ（`DDS-PF` / `DDS-DSPF` / `DDS-PRTF`）。種別は拡張子で決まる。
- 原典の書き方は種別ごとに揺れる。桁の抽出では次を正規化している（`generate-dds-columns.mjs`）:
  区切りが「から」「-」「−」と混在／文型が2通り／表示装置は先頭3欄を「1-7」とだけ書く／
  **表示装置と印刷装置は条件付けを「7-16」と注記桁を含めて書く**（物理/論理は「8-16」）。
  最後のものは注記(7桁)と衝突して条件付けの欄が消えるため、8 桁目始まりに揃える。
- キーワードの抽出（`generate-dds-keywords.mjs`）は **`<li>` のクラス名に依存しない**こと。
  日本語版は `ulchildlink` が付くが英語版は付かない。キーワード詳細ページへのリンクを起点に
  次のリンクの手前までを1項目として切り出す。項目をまたぐと隣のキーワードの説明が付く。

### DDS の F4 プロンプター（定位置項目）

- 桁と欄名は **navigation の桁定義を使い回す**（原典から `generate-dds-columns.mjs` が
  作ったもの）。原典の書き方の揺れの吸収はそちらに集約されており、作り直すと同じ罠を
  二度踏む。プロンプターとルーラーで桁が食い違わない担保にもなる（`verify-dds-prompter.mjs`）。
- **有効な値は欄ごとの詳細ページにある**。概要ページには無い。並べ方が 2 通りあり、
  物理/論理・表示装置は定義リスト（`<dt>B</dt><dd>…</dd>`）、印刷装置は表
  （`項目`/`意味`）。片方だけ読むと印刷装置の選択欄が丸ごと落ちる。
- 欄と詳細ページの対応は **桁で突き合わせる**。並び順で対応付けると、欄の数と詳細ページの
  数が合わないとき（表示装置は 14 欄に対し 12 ページ）に 1 つずつずれて別の欄の説明が付く。
  桁の書き方は日英で語順が逆（ja `(7 - 16 桁目)` / en `(positions 7 through 16)`）なので、
  括弧内の数字の最小・最大を範囲として扱う。
- **長さの欄(30-34)は右寄せが必須**。原典に明記があり（ja「右寄せ」/ en「right-aligned」）、
  実機でも左詰めは `CPF7311` で作成できない。`attributes.numericOnly` を付けて
  書き戻しを `padStart` にする。データ・タイプの `5`(2進文字) のように、知識で
  切り捨てそうな値も原典にはあるので、値は原典の一覧をそのまま採る。

### RPG の補完（命令コード・組み込み関数・仕様書キーワード）

- 何を出すかは **「どの仕様書の、どの桁にいるか」** で決まる。C 仕様の命令コード欄(26-35)は
  命令コード、H/F/D/P 仕様の 44 桁目以降はその仕様書のキーワード、`%` の直後は組み込み関数。
  仕様書の判定は `specClassifier` に集約する（ルーラー・プロンプターと共有）。
- 原典の書き方が対象で違う。**仕様書キーワードは索引の表題そのものが構文**
  （`ACTGRP(*STGMDL | *NEW | …)`）だが、**命令・関数は詳細ページ側**にある。
- 抽出時の罠（いずれも実際に踏んだ）:
  - `ANDxx` / `IFxx` / `DOWxx` / `ENDyy` は比較演算子の位置を**小文字**で表す。
    名前を大文字だけに限ると 22 件落ちる。
  - 旧命令（MOVEL/ADD 等）の「自由形式構文」欄には
    「(許可されていない - EVAL を使用)」という**注記**が入る。構文ではないので
    命令名で始まるかを見て弾く。
  - 組み込み関数の詳細ページは表を持たず、本文に構文がある。ただし本文には
    使用例（`%date(string:*MDY0)` のように小文字）も出るため、関数名と同じ表記の
    ものだけ採る。表題（`%SUBST (サブストリングの検索)`）を構文として拾う事故もある。
- **方言(ILE / RPG III)で桁も語彙も違う**。RPG III は命令コード欄が **28-32 桁**（ILE は 26-35）で、
  5 文字しか入らない。そのため綴りが別物になる（`LOOKUP`→`LOKUP`、`EXCEPT`→`EXCPT`、`CHECKR`→`CHEKR`）。
  ILE の一覧を桁数で絞っても RPG III の語彙にはならない。組み込み関数と `EVAL`/`CALLP`/`IF`/`DOW` 等は
  RPG IV で入ったもので RPG III には無い。RPG III の H/F 仕様は 71 桁目まで固定欄で、
  ILE のようなキーワード欄が無い（`prompter/rpg/rpg3/F-SPEC.json` 参照）。
- **RPG III の命令集合の原典は実機のコンパイラ**。IBM の RPG/400 Reference は ibm.com/docs に生 HTML が
  無く PDF も入手できず、`docs/origin/rpg3/` の第三者資料は System/3 系 RPG II/III のもので足りない。
  1命令1行の C 仕様を `CRTRPGPGM OPTION(*SOURCE) GENLVL(50)` に通し、**QRG5014『Operation entry invalid』**が
  付いた行を弾く（`probe-rpg3-opcodes.sh`）。判定時の注意:
  - **対照として存在しない命令(`ZZZZZ`)を必ず混ぜる**。これが QRG5014 にならなければ手法が壊れている。
  - `BEGSR`/`ENDSR` は後続行を巻き添えにする(QRG5005『Operation entry following ENDSR operation invalid』)。
    本体から外し、正しい副ルーチンの形で別に測る。外さないと後続が検査されず全部「有効」に見える。
  - **候補に無い命令は検出できない**。実際 `CHEKR` を当初落としており、追試で拾った。
- **日英で件数が一致しない**。原典そのものの差で、新しめの命令・関数
  （`%CONCAT` / `FOR-EACH` 等）は日本語の翻訳が無い。検査でも件数一致は求めず、
  各言語の索引に対する取りこぼしだけを見る。

### sh / ps1 二本立て CLI の実装観点（既知の罠）

`.claude/skills/aidev-docs/bin/aidev`（POSIX sh・正本）と `.claude/skills/aidev-docs/bin/aidev.ps1`（PowerShell）は**挙動・出力・終了コードを一致**
させる二本立て。実装・レビュー時は、両言語で踏みやすい次の罠を点検する（過去に実際に踏んでいる）。

- **POSIX sh は `set -eu` 前提**。次の2点に注意する。
  - **列挙/探索を行う関数は末尾に `return 0` を置く**。ループ最後の比較が偽(1)を返すと関数が非0終了し、
    `var=$(func)` の**代入ごとスクリプトが停止**する（無言 exit 1 になりやすい）。
    例: `works_matching_slug` が最後の `[ "$x" = "$slug" ]` の偽値を返して `matches=$(...)` で落ちた。
  - 一般に `var=$(cmd)` は cmd の非0終了で停止しうる。**意図的に非0を許すなら `|| true` で受ける**。
    なお `&&`/`||` リストの非末尾・`if`/`while`/`until` の条件部は `set -e` の対象外（exempt）。
- **PowerShell の単一要素配列アンラップに注意**。要素1個の配列は `return` で**スカラーにアンロール**され、
  `$x[0]` が（配列の先頭要素ではなく）**文字列の先頭1文字**を返す。
  - 配列を期待する呼び出し側は **`$x = @(Func ...)` で配列強制**する（0件→Count0 / 1件→[0]正 / 複数→正）。
    例: `Wt-Add` の `$mw` が1件一致時に先頭1文字化し、worktree の `.aidev/current` を壊した（`@()` で修正）。
- 検証の担保: sh⇔ps1 の差異は `.claude/skills/aidev-docs/bin/test/run.sh` のパリティ節（pwsh 環境/CI）で機械的に突合する。
